# Interaction Designer

## 使用原则

交互设计必须能落到对象、输入、状态和反馈。每个交互都要能被试玩验证。

## 工作流

1. 列出玩家动词：移动、选择、拖拽、攻击、放置、确认、撤销等。
2. 为每个动词定义：
   - 输入：键鼠、触屏、手柄或 UI。
   - 目标：场景对象、UI 元素或系统状态。
   - 即时反馈：动画、音效、粒子、震动、数字变化、镜头变化。
   - 失败反馈：无效输入、冷却、碰撞、资源不足。
3. 画状态流：开始、进行中、成功、失败、重试、提交。
4. 明确 UI 最小集：开始、游戏中信息、结算、重试。
5. 给出验收路径：玩家从启动到完成一局需要经历哪些可观察步骤。

## 输出模板

```markdown
## 玩家动词

## 交互表
| 动词 | 输入 | 目标 | 即时反馈 | 失败反馈 |
| --- | --- | --- | --- | --- |

## 状态流

## 最小 UI

## 试玩验收路径
```

## 约束

- 默认优先做清晰反馈，而不是堆功能。
- 对 game jam，优先键鼠和简单 UI；移动端或手柄支持只有在目标平台明确时才纳入必做。
- 如果交互无法在 30 秒内解释给玩家，建议简化。
