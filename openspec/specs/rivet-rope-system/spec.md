## Requirements
### Requirement: 铆钉和绳索系统职责边界

铆钉和绳索系统 MUST 作为独立玩法规则能力，负责玩法铆钉库存、插锚、拔锚、绳索路径、绳长约束、坠落保护和伤害结果输出。系统 MUST NOT 负责玩家输入映射、手部移动、抓点稳定判定、耐力脱力判定、关卡抓点生成、角色动画表现或生命值最终扣除。

#### Scenario: 消费受力系统坠落事实

- **WHEN** 受力系统输出某名玩家进入坠落的事实
- **THEN** 铆钉和绳索系统 MUST 基于当前铆钉、绳索和玩家位置计算保护结果，并 MUST NOT 重新判定该玩家是否因抓点或耐力而坠落

#### Scenario: 输出伤害结果而不扣生命

- **WHEN** 坠落结算产生伤害
- **THEN** 铆钉和绳索系统 MUST 输出结构化伤害结果，并 MUST NOT 直接修改玩家生命值

### Requirement: 玩法铆钉与关卡抓点分离

系统 MUST 将玩家携带和部署的玩法铆钉与关卡系统中的可抓取 `AnchorPoint` 区分管理。玩法铆钉 SHALL 使用独立运行时状态记录铆钉 ID、携带者或部署者、部署位置、当前处于携带还是已部署、是否参与当前绳索路径以及调试信息。

#### Scenario: 关卡抓点不自动成为保护铆钉

- **WHEN** 关卡中存在可抓取 `AnchorPoint`
- **THEN** 铆钉和绳索系统 MUST NOT 将该 `AnchorPoint` 自动视为玩家已经部署的保护铆钉

#### Scenario: 玩家部署后生成玩法铆钉状态

- **WHEN** 玩家在合法位置成功插锚
- **THEN** 系统 MUST 创建或激活一个玩法铆钉状态，并记录其世界位置、部署者、铆钉 ID 和已部署状态

### Requirement: 铆钉库存守恒

系统 MUST 维护可配置的总铆钉数量，MVP 默认值 SHALL 为 5。任意时刻所有铆钉 MUST 只处于某名玩家携带或场上已部署两类核心状态之一，且总数量 MUST 保持守恒。玩家 MUST 只能部署自己携带的铆钉，系统 MUST NOT 允许两个玩家远程直接互换库存。

#### Scenario: 初始化铆钉总数

- **WHEN** 关卡初始化铆钉和绳索系统
- **THEN** 系统 MUST 根据配置初始化总铆钉数量，并使所有玩家携带数量和场上部署数量之和等于总数

#### Scenario: 无库存时不能插锚

- **WHEN** 某名玩家当前携带铆钉数量为 0
- **THEN** 该玩家尝试插锚 MUST 失败，并且场上部署数量 MUST NOT 增加

#### Scenario: 不允许远程转移铆钉

- **WHEN** 两名玩家没有通过回收或关卡规则触发库存变化
- **THEN** 系统 MUST NOT 将一名玩家携带的铆钉直接转移给另一名玩家

### Requirement: 插锚规则

系统 MUST 支持玩家在合法墙面或合法可插锚区域插锚。插锚请求 MUST 至少包含玩家标识、候选世界位置、候选表面是否合法、当前携带库存和当前绳索状态。成功插锚后，系统 MUST 消耗该玩家 1 个携带铆钉并将新铆钉加入绳索候选路径。

#### Scenario: 合法位置插锚成功

- **WHEN** 玩家拥有至少 1 个携带铆钉，且候选位置被交互适配器标记为可插锚
- **THEN** 系统 MUST 消耗该玩家 1 个携带铆钉，并在候选位置部署一个玩法铆钉

#### Scenario: 非法位置插锚失败

- **WHEN** 玩家尝试在未命中墙面、不可插锚区域或其他非法候选位置插锚
- **THEN** 系统 MUST 拒绝插锚，并保持玩家携带数量和场上铆钉状态不变

#### Scenario: 插锚后参与绳索路径

- **WHEN** 一个玩法铆钉成功部署并处于场上已部署状态
- **THEN** 系统 SHALL 将其纳入后续绳索路径计算的候选集合

### Requirement: 拔锚与回收规则

系统 MUST 支持玩家回收场上已部署铆钉。只要玩家靠近已部署铆钉且自身处于 stable 状态，交互层 SHALL 显示回收按钮；玩家点击后，系统 MUST 直接回收该铆钉。回收请求 MUST 至少包含玩家标识、目标铆钉 ID、玩家与目标的距离和玩家 stable 状态。成功回收后，目标铆钉 MUST 从场上部署状态移除，并将 1 个铆钉加入回收玩家的携带库存。

#### Scenario: 后方玩家回收铆钉

- **WHEN** 后方玩家处于目标铆钉回收范围内且自身处于 stable 状态
- **THEN** 交互层 SHALL 显示回收按钮，玩家点击后系统 MUST 回收该铆钉，从绳索路径候选集合移除它，并将 1 个铆钉加入该玩家携带库存

#### Scenario: 回收正在参与绳路的铆钉立即重算绳路

- **WHEN** 玩家回收的目标铆钉正在参与当前绳索路径或作为坠落保护点
- **THEN** 系统 MUST 直接回收该铆钉，并立即用剩余已部署铆钉重算绳索路径、保护点、估算坠落距离和伤害结果

#### Scenario: 超出范围不可回收

- **WHEN** 玩家与目标铆钉距离大于配置的回收范围
- **THEN** 系统 MUST 拒绝回收该铆钉

#### Scenario: 非 stable 状态不可回收

- **WHEN** 玩家靠近目标铆钉但自身不处于 stable 状态
- **THEN** 交互层 MUST NOT 显示可点击的回收按钮，系统 MUST 拒绝回收该铆钉

### Requirement: 打钉和回收事件同步

系统 MUST 在打钉成功和回收成功后通过 `game.event` 同步事件，使远端客户端能更新已部署铆钉列表、玩家库存显示和绳索路径表现。打钉成功事件 SHALL 使用 `rivet.place` 事件类型，回收成功事件 SHALL 使用 `rivet.collect` 事件类型。事件 payload MUST 至少包含事件 ID、操作者玩家 ID、铆钉 ID、操作后的玩家携带库存，以及绳路版本号或等价递增序号。打钉事件 MUST 包含部署位置；回收事件 MUST 包含被回收铆钉 ID。服务器 MUST 只做 `game.event` 基础校验和转发，不理解或重算铆钉规则。

#### Scenario: 打钉成功发送同步事件

- **WHEN** 玩家成功部署一个铆钉
- **THEN** 本地客户端 MUST 发送 `game.event`，其 `eventType` MUST 为 `rivet.place`，并包含铆钉 ID、部署位置、操作者玩家 ID、操作后库存和绳路版本号

#### Scenario: 回收成功发送同步事件

- **WHEN** 玩家成功回收一个已部署铆钉
- **THEN** 本地客户端 MUST 发送 `game.event`，其 `eventType` MUST 为 `rivet.collect`，并包含被回收铆钉 ID、操作者玩家 ID、操作后库存和绳路版本号

#### Scenario: 远端去重并更新绳路

- **WHEN** 远端客户端收到 `rivet.place` 或 `rivet.collect` 事件
- **THEN** 远端客户端 MUST 按 `eventId` 去重，并根据事件更新只读铆钉列表、玩家库存显示和绳索路径表现

#### Scenario: 失败尝试不同步

- **WHEN** 玩家尝试打钉或回收但本地规则判定失败
- **THEN** 客户端 MUST NOT 发送 `rivet.place` 或 `rivet.collect` 成功事件

### Requirement: 主玩法正式 UI 入口

主玩法场景 SHALL 提供正式 UGUI 操作入口，用于提交插锚、回收铆钉和收绳救援请求。正式 UI MUST 只响应玩家点击按钮，不得额外监听键盘、鼠标按键或触摸区域来触发调试行为。按钮可见状态 MUST 根据当前库存、附近可回收铆钉和救援窗口刷新；不满足条件的插锚、回收或收绳救援入口 MUST NOT 常驻显示。

#### Scenario: 插锚按钮提交插锚请求

- **WHEN** 当前领攀玩家拥有携带铆钉且主玩法 UI 中点击插锚按钮
- **THEN** UI SHALL 向铆钉和绳索系统提交插锚请求，并使用当前候选插锚位置作为请求位置

#### Scenario: 回收按钮只在可回收时启用

- **WHEN** 玩家附近存在已部署铆钉且玩家处于 stable 状态
- **THEN** 主玩法 UI SHALL 显示并启用回收按钮，点击后提交回收请求

#### Scenario: 不可回收时隐藏回收入口

- **WHEN** 玩家附近不存在可回收铆钉，或玩家不处于 stable 状态
- **THEN** 主玩法 UI MUST NOT 显示回收按钮或回收提示信息

#### Scenario: 收绳按钮只在救援窗口内有效

- **WHEN** 救援窗口处于开启状态
- **THEN** 主玩法 UI SHALL 显示并启用收绳按钮，点击后提交一次收绳救援请求

#### Scenario: 非救援窗口隐藏收绳入口

- **WHEN** 救援窗口未开启
- **THEN** 主玩法 UI MUST NOT 显示收绳救援按钮、收绳量或救援提示信息

### Requirement: 领攀与跟攀协作

系统 SHALL 支持记录当前领攀玩家和跟攀玩家。当前领攀玩家负责向上部署铆钉，跟攀玩家在经过或接近已部署铆钉后回收铆钉。系统 MUST 支持在安全状态下切换领攀关系，并 MUST 保持铆钉库存归属于实际携带或回收的玩家。

#### Scenario: 领攀玩家部署铆钉

- **WHEN** 当前领攀玩家在合法位置插锚且拥有携带铆钉
- **THEN** 系统 SHALL 允许其部署铆钉，并记录部署者为当前领攀玩家

#### Scenario: 跟攀玩家回收后获得库存

- **WHEN** 当前跟攀玩家成功回收一个已部署铆钉
- **THEN** 系统 MUST 将该铆钉加入跟攀玩家携带库存，而不是加入远端领攀玩家库存

#### Scenario: 安全状态下切换领攀

- **WHEN** 两名玩家均未处于坠落结算或不可交互状态，且协作流程触发换领
- **THEN** 系统 SHALL 更新领攀/跟攀身份，并保持两名玩家各自携带库存不变

### Requirement: 绳索路径计算

系统 MUST 根据两名玩家端点和已部署铆钉计算逻辑绳索路径。路径中的铆钉 MUST 按关卡攀爬进度或等价排序方式排列，已回收或不在当前绳索链路中的铆钉 MUST NOT 参与路径长度计算。

#### Scenario: 双人绳索端点绑定到腰部

- **WHEN** 主玩法场景中存在本地 3C 角色和远端角色
- **THEN** 表现层 SHALL 将绳索两端绑定到两名角色的腰部挂点或等价腰部偏移位置
- **AND** 腰部挂点更新 MUST NOT 直接移动角色 Transform、手部目标点或镜头状态

#### Scenario: 视觉绳索避免从角色正前方穿插

- **WHEN** 表现层在主玩法场景中渲染玩家腰部到铆钉之间的绳段
- **THEN** 表现层 SHALL 允许对腰部挂点和插锚表现点应用墙面方向或角色侧边的视觉偏移，避免绳索从角色正前方穿插身体
- **AND** 该视觉偏移 MUST NOT 改变库存、回收范围、坠落保护或伤害判定

#### Scenario: 已部署铆钉进入路径

- **WHEN** 两名玩家之间存在一个已部署铆钉，且该铆钉位于当前绳索链路中
- **THEN** 系统 MUST 将绳索路径表示为玩家端点经过该铆钉再到另一名玩家端点的逻辑折线

#### Scenario: 回收铆钉退出路径

- **WHEN** 一个已部署铆钉被成功回收
- **THEN** 系统 MUST 在下一次绳索路径计算中排除该铆钉

#### Scenario: 无已部署铆钉时直连玩家

- **WHEN** 当前绳索链路中没有任何已部署铆钉
- **THEN** 系统 MUST 将绳索路径视为两名玩家端点之间的直接路径

### Requirement: 绳长约束

系统 MUST 维护可配置的绳索总长度，MVP 默认值 MUST 为 20 米或等价 Unity 单位。绳索总长度 MUST NOT 因插锚而增加；成功插锚只会为绳索新增一个穿过点或路径转折点。系统 MUST 基于逻辑绳索路径计算已占用长度、剩余松弛量和绳索状态。绳索状态 MUST 至少区分松弛和拉直。系统 MUST 输出绳长约束结果供角色 3C 消费，但 MUST NOT 直接移动角色手部或镜头。

#### Scenario: 初始化 20 米绳长

- **WHEN** 关卡未覆盖绳索总长度配置
- **THEN** 系统 MUST 使用 20 米或等价 Unity 单位作为默认绳索总长度

#### Scenario: 路径长度小于绳长时松弛

- **WHEN** 当前逻辑绳索路径长度小于配置的绳索总长度
- **THEN** 系统 MUST 输出绳索松弛状态，并提供剩余松弛量

#### Scenario: 路径长度达到绳长时拉直

- **WHEN** 当前逻辑绳索路径长度达到或超过配置的绳索总长度
- **THEN** 系统 MUST 输出绳索拉直状态，并提供超出长度或需要约束的距离

#### Scenario: 拉直状态不直接移动角色

- **WHEN** 绳索状态变为拉直
- **THEN** 系统 MUST 向角色 3C 输出约束结果，并 MUST NOT 直接修改玩家 Transform、手部目标点或镜头状态

#### Scenario: 插锚不增加总绳长

- **WHEN** 玩家成功部署一个新铆钉
- **THEN** 系统 MUST 将该铆钉加入逻辑绳索路径候选集合，并 MUST 保持绳索总长度不变

#### Scenario: 多余绳长表现为松弛

- **WHEN** 当前逻辑绳索路径长度小于绳索总长度
- **THEN** 系统 SHALL 为表现层提供剩余松弛量，使表现层能将多余绳长显示为绳段下垂、下方玩家身边绳圈或等价 UI 提示

#### Scenario: 视觉绳索具有动态松弛表现

- **WHEN** 表现层渲染处于松弛状态的绳索路径
- **THEN** 表现层 SHALL 使用多段渲染、端点平滑、横向弧线、轻微下垂或小幅摆动等方式避免绳索像硬杆一样僵直
- **AND** 这些动态表现 MUST NOT 参与绳长、保护点、坠落距离或伤害判定

### Requirement: 坠落保护判定

当受力系统触发某名玩家坠落时，铆钉和绳索系统 MUST 判定是否存在可保护该坠落者的最高已部署保护铆钉。没有保护铆钉时，系统 MUST 输出无保护坠落。有保护铆钉时，系统 MUST 基于绳索总长度、保护者到保护铆钉的路径占用长度、保护铆钉到坠落者的路径长度和剩余松弛量计算估算自由坠落距离。

#### Scenario: 无已部署保护铆钉

- **WHEN** 玩家坠落且当前绳索链路中不存在可保护该坠落者的已部署铆钉
- **THEN** 系统 MUST 输出无保护坠落结果，并标记没有保护铆钉

#### Scenario: 使用最高已部署保护铆钉

- **WHEN** 玩家坠落且当前绳索链路中存在多个可保护该坠落者的已部署铆钉
- **THEN** 系统 MUST 选择按攀爬进度最高的已部署保护铆钉参与坠落保护计算

#### Scenario: 计算受保护坠落距离

- **WHEN** 玩家坠落且存在已部署保护铆钉
- **THEN** 系统 MUST 输出受保护坠落结果，并提供估算自由坠落距离、绳索是否拉直和使用的保护铆钉 ID

### Requirement: 下方玩家主动收绳减伤

当上方玩家触发坠落时，系统 SHALL 为下方玩家开启一个可配置的收绳救援窗口。下方玩家在救援窗口内快速点击收绳输入时，系统 MUST 校验下方玩家是否处于 stable 且可交互状态，并将有效点击转换为收绳进度、等价收绳长度或救援减伤倍率。收绳效果 MUST 具有最大上限，并 MUST NOT 将最终伤害降低到配置的最小伤害以下。

#### Scenario: 坠落时开启救援窗口

- **WHEN** 上方玩家触发坠落，且下方玩家处于 stable、未坠落且可交互状态
- **THEN** 系统 SHALL 开启收绳救援窗口，并允许下方玩家提交收绳点击输入

#### Scenario: 有效点击减少坠落伤害

- **WHEN** 下方玩家在救援窗口内提交有效收绳点击
- **THEN** 系统 MUST 增加本次坠落结算的收绳进度，并在伤害计算中降低估算自由坠落距离或额外降低伤害倍率

#### Scenario: 收绳效果存在上限

- **WHEN** 下方玩家持续快速点击收绳直到达到配置的最大收绳效果
- **THEN** 系统 MUST 停止继续增加本次坠落结算的收绳减伤收益，并保持最终伤害不低于配置的最小伤害

#### Scenario: 不可救援状态下点击无效

- **WHEN** 下方玩家处于坠落、脱力、受击或不可交互状态
- **THEN** 系统 MUST 拒绝本次收绳点击，并 MUST NOT 增加收绳进度或救援减伤倍率

#### Scenario: 救援窗口结束后点击无效

- **WHEN** 坠落结算窗口已经结束、坠落者落定或绳索恢复非救援状态
- **THEN** 后续收绳点击 MUST NOT 影响该次坠落伤害结果

### Requirement: 伤害结果计算

系统 MUST 根据坠落保护结果和主动收绳结果生成结构化伤害结果。伤害结果 MUST 至少包含坠落者 ID、是否受保护、保护铆钉 ID、估算自由坠落距离、有效收绳量、救援减伤倍率、伤害倍率、建议伤害值和主要原因。对于相同保护状态和相同收绳效果，估算自由坠落距离越大，建议伤害值 MUST 不小于较短距离的伤害值。

#### Scenario: 受保护坠落降低伤害

- **WHEN** 两次坠落的估算自由坠落距离相同，但一次存在已部署保护铆钉，另一次没有已部署保护铆钉
- **THEN** 受保护坠落的建议伤害值 MUST 小于或等于无保护坠落的建议伤害值

#### Scenario: 坠落距离增加伤害不降低

- **WHEN** 同一保护状态下发生两次坠落，第二次估算自由坠落距离大于第一次
- **THEN** 第二次输出的建议伤害值 MUST 大于或等于第一次

#### Scenario: 有效收绳降低伤害

- **WHEN** 两次坠落的保护状态和估算自由坠落距离相同，但第二次存在更高的有效收绳量
- **THEN** 第二次输出的建议伤害值 MUST 小于或等于第一次

#### Scenario: 伤害结果包含调试原因

- **WHEN** 系统输出坠落伤害结果
- **THEN** 结果 MUST 包含主要原因，用于区分无锚保护、绳长不足、受保护拉直、主动收绳减伤或其他可调试状态

### Requirement: 参数配置和调试输出

铆钉和绳索系统 MUST 提供可调参数配置和调试输出。参数配置 MUST 至少包含总铆钉数量、绳索总长度、插锚范围、回收范围、收绳救援窗口时长、单次点击收绳量、每秒有效点击上限、最大收绳减伤上限、最小坠落伤害、受保护伤害倍率、无保护伤害倍率和伤害曲线。调试输出 MUST 至少包含两名玩家携带库存、场上铆钉列表、当前领攀关系、绳索路径长度、剩余松弛量、绳索状态、救援窗口状态、有效收绳量和最近一次坠落结算结果。

#### Scenario: 运行时调整参数

- **WHEN** 开发者调整绳索总长度、插锚范围、回收范围或伤害倍率配置
- **THEN** 系统 MUST 在后续插拔锚、绳长约束和坠落结算中使用更新后的参数

#### Scenario: 查看调试状态

- **WHEN** 开发者在测试场景中启用铆钉和绳索调试
- **THEN** 系统 MUST 显示或记录库存、场上铆钉、绳索状态、剩余松弛量、救援窗口状态、有效收绳量和最近一次坠落结算结果

### Requirement: MVP 验收场景

铆钉和绳索系统 MUST 提供可在 Unity/Tuanjie 项目中验证的 MVP 验收路径，覆盖无锚坠落、有锚保护、主动收绳减伤、绳长拉直、插锚消耗、拔锚回收、领攀交替和伤害随坠落距离变化。

#### Scenario: 插拔锚库存验收

- **WHEN** 测试玩家连续执行合法插锚、合法回收和非法插锚
- **THEN** 系统 MUST 正确更新携带库存、场上铆钉数量和失败原因，并保持总铆钉数量守恒

#### Scenario: 绳索保护验收

- **WHEN** 测试玩家分别在无保护铆钉和有保护铆钉两种状态下触发坠落
- **THEN** 系统 MUST 分别输出无保护坠落和受保护坠落结果，并给出可比较的建议伤害值

#### Scenario: 主动收绳验收

- **WHEN** 上方玩家触发坠落且下方玩家在救援窗口内快速点击收绳
- **THEN** 系统 MUST 记录有效收绳量，并输出低于未收绳对照情况的建议伤害值

#### Scenario: 绳长约束验收

- **WHEN** 两名玩家沿攀爬方向逐渐拉开距离直到逻辑绳索路径长度达到配置绳长
- **THEN** 系统 MUST 从松弛状态切换到拉直状态，并输出可供角色 3C 消费的约束距离
