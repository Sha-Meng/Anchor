## Context

Anchor 的核心玩法是双人协作登山，玩家主要操控布娃娃的左右手在崖壁上摸索、吸附、脱力和坠落。当前策划文档已经拆出角色 3C、关卡系统、受力系统、铆钉和绳索系统四个职责，但受力系统尚未定义清楚：它既不能过度侵入角色手感，也不能把绳索伤害和关卡事实混在一起。

受力系统需要在 Tuanjie Editor 1.9.3 / Unity 2022.3.62t11 中以 game jam 可控成本实现。MVP 应优先支持可读、可调、可调试的规则判定，而不是追求完整真实物理。

## Goals / Non-Goals

**Goals:**

- 定义一个独立的受力判定服务/组件，接收外部系统提供的快照输入并输出离散受力状态。
- 支持稳定、左手脱力、右手脱力、双手脱力、坠落等核心结果，供角色 3C 和铆钉绳索系统消费。
- 支持攀附点、障碍、假点、耐力不足和身体状态对判定的影响。
- 提供可调参数和调试输出，方便在 jam 阶段快速调整手感。
- 保持受力系统与关卡生成、角色输入、绳索伤害计算解耦。

**Non-Goals:**

- 不实现完整布娃娃物理模拟或真实力学积分。
- 不负责玩家输入、手部移动、镜头、震动反馈和动画表现。
- 不负责铆钉库存、绳长约束、坠落伤害或生命扣除。
- 不要求服务器权威判定；MVP 继续由本地客户端计算自身角色状态。

## Decisions

### Decision 1: 使用快照输入和离散结果输出

受力系统采用 `ForceEvaluationInput` 风格的数据快照，包含左右手当前抓取状态、命中的点类型、抓点稳定值、障碍/假点状态、耐力状态、身体是否已经处于坠落/受击等信息。输出采用 `ForceEvaluationResult` 风格的结果，至少包含整体状态、左右手是否有效受力、脱力原因和是否触发坠落。

理由：快照输入能让角色 3C、关卡系统和测试代码都用同一套接口驱动判定，避免受力系统直接读取场景对象导致耦合。离散结果便于 UI、日志、调试面板和联网 payload 后续复用。

备选方案：直接把判定写进角色控制器中。该方案短期快，但会让角色手感、关卡点类型和绳索触发交织在一个脚本里，后续调参和并行开发成本更高。

### Decision 2: MVP 使用规则表而非完整物理求解

MVP 判定按优先级执行：非法接触或障碍先使对应手无效，假点在暴露后使对应手无效，抓点稳定值低于最小抓附阈值时使对应手无效，耐力耗尽使对应手脱力，最后根据有效受力手数量和缓冲计时判断稳定、单手脱力、双手脱力或坠落。必要时可加入身体姿态阈值，例如身体离崖壁过远或旋转过大时触发不稳定。

理由：game jam 目标是快速得到可玩手感。规则表能清晰表达策划意图，便于设计师调参数，也能通过编辑器日志和简单测试验证。

备选方案：基于 Rigidbody/Joint 受力持续模拟并由物理结果自然决定脱力。该方案表现上可能更真实，但调参难、测试难，且容易把系统目标变成“调物理 bug”。

### Decision 3: 受力系统只发布坠落事实，不计算绳索保护和伤害

当判定结果进入坠落时，受力系统只输出“角色已失去有效受力”的事实和原因。铆钉和绳索系统再根据绳长、最上方有效锚钉、角色位置计算保护和伤害。

理由：受力系统关注“为什么掉”，绳索系统关注“掉了以后是否被保护以及伤害多少”。两者分开后，角色 3C 可以只关心表现状态，绳索系统也能独立测试。

备选方案：受力系统直接计算掉落距离和伤害。该方案会把锚钉库存、绳长和生命规则提前绑死，不利于后续关卡和数值迭代。

### Decision 4: 提供调试可视化和参数资产

实现阶段应提供一个可编辑参数配置，至少包含耐力阈值、最小抓附稳定值、稳定抓附阈值、低稳定值耐力消耗倍率、假点暴露后的失效规则、单手稳定容忍时间、双手失效到坠落的缓冲时间，以及调试日志开关。调试输出应能显示左右手有效性、当前状态、主要失败原因、抓点稳定值和最近一次状态切换。

理由：受力手感需要反复调试。把参数和结果暴露出来，比把规则硬编码在脚本里更适合 jam 期间快速试错。

备选方案：全部常量硬编码。该方案开发最快，但一旦手感不好，需要程序反复改代码，影响迭代速度。

### Decision 5: 关卡查询、适配器和评估器分层

关卡系统通过 `IGripQueryProvider` 风格接口提供抓点事实，输入是手部世界位置和查询半径，输出是 `GripQueryResult` 风格数据：点位类型、稳定值、假点是否已暴露、点位标识、表面法线和可选调试名。关卡系统 MUST NOT 返回角色最终是否稳定，因为稳定还取决于耐力、另一只手、身体状态和状态缓冲。

`ForceInputAdapter` 负责从角色 3C、关卡查询和耐力系统收集一帧事实，组装为 `ForceEvaluationInput`。`ForceEvaluator` 只消费该输入、上一帧少量计时状态和参数配置，返回 `ForceEvaluationResult`。单元测试可以跳过关卡和角色系统，直接构造 `ForceEvaluationInput` 覆盖所有规则组合。

理由：该分层让其他系统未完成时也能先完成受力系统的核心单元测试；其他模块完成后只需实现或接入查询接口和适配器，不需要改动核心计算。

备选方案：由受力系统直接调用场景 Collider 或关卡对象查询。该方案接入直观，但会让评估器依赖 Unity 场景，难以单元测试，也不适合高频调用。

## Risks / Trade-offs

- [Risk] 规则判定过于离散，可能让角色表现显得“突然脱力” → Mitigation：为单手脱力和双手脱力加入短缓冲时间，并由角色 3C 负责抖动、滑落、动画和震动过渡。
- [Risk] 受力系统输入字段过多，早期实现变慢 → Mitigation：MVP 先实现左右手接触、点类型、抓点稳定值、耐力和身体状态五类输入，障碍/假点细节通过枚举扩展。
- [Risk] 关卡查询过于频繁影响性能 → Mitigation：场景查询放在适配器侧，并允许缓存手部当前命中的候选点；评估器只读取快照，不做物理查询和分配。
- [Risk] 与布娃娃物理实际表现不一致 → Mitigation：受力系统输出作为玩法状态源，物理表现跟随状态；真实物理只作为视觉和碰撞辅助。
- [Risk] 双人联机状态可能出现本地判定差异 → Mitigation：MVP 沿用客户端自权威角色状态，远端只显示同步结果，不在服务器重算受力。

## Migration Plan

1. 在 OpenSpec 中新增 `force-system` 能力规格。
2. 实现阶段先新增纯 C# 判定模型和单元/编辑器测试，不改动现有场景行为。
3. 定义关卡抓点查询接口和受力输入适配器，将手部接触、抓点事实和耐力状态转换为受力输入。
4. 在测试场景中启用调试面板或日志验证稳定、单手脱力、双手脱力和坠落状态。
5. 若接入后手感不符合预期，可只替换规则评估器或参数配置，不回滚角色 3C 和绳索系统边界。
