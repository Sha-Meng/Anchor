## Context

Anchor 的核心攀爬体验依赖大量高频抓点查询：手部摸索需要知道附近是否有真实抓点，受力系统需要稳定值和点位事实，震动反馈需要距离信息，调试工具需要可视化当前命中的抓点。当前玩法文档已经明确 Gym 是主关卡，挂有 `Anchor Point` 脚本的节点代表真实抓取点，但还没有定义这些数据在 Gameplay 初始化后如何管理、如何查询、稳定值如何解释。

本设计补齐关卡系统内部的数据模型和查询边界。关卡系统只回答“地图上有什么、离哪里最近、某个世界点在当前关卡事实下稳定值是多少”，不回答“角色最终是否稳定”。最终稳定仍由受力系统结合左右手、耐力和身体状态判断。

## Goals / Non-Goals

**Goals:**

- 定义 Gameplay 初始化时的关卡抓点注册、校验和运行时数据快照。
- 定义 `Anchor Point` 作为真实抓取点的数据字段和稳定值语义。
- 提供高频调用友好的基础接口：查询某点稳定值、查询最近抓点、可选查询半径内候选点。
- 让查询结果可被角色 3C、受力系统、震动反馈和调试工具共同消费。
- 给出稳定值判定和数据管理的推荐方案、备选方案与待讨论问题。

**Non-Goals:**

- 不在关卡系统中实现角色最终受力状态、脱力状态或坠落状态。
- 不在本变更中实现完整关卡编辑器、自动生成攀爬路线或动态地形破坏。
- 不负责障碍点、假点、生命扣除或绳索保护的完整规则；这些可在后续关卡事实类型中扩展。
- 不引入第三方空间索引或寻路库。

## Decisions

### Decision 1: Gameplay 初始化时生成关卡抓点运行时快照

Gym 场景中每个启用的 `Anchor Point` 组件在 Gameplay 初始化阶段被 `LevelAnchorRegistry` 收集，转换为轻量运行时数据，例如 `AnchorPointData`：运行时分配的抓点 ID、世界坐标、手动配置的基础稳定值、直接沿用现有 `previewSlightRadius` 的抓取半径、来自 `gameObject.name` 的调试名、是否启用。初始化结束后，查询服务只读取这份数据快照和空间索引。

理由：高频查询不应每次访问场景层级、组件字段或 `FindObjectsOfType`。运行时快照可以集中做校验、归一化和索引构建，也能让调试工具显示“当前关卡实际可查询的数据”。

备选方案：查询时直接遍历场景中的 `Anchor Point` 组件。该方案开发最快，但手部查询频率高时容易产生性能问题，也会让运行期组件启停、层级变更和查询行为耦合。

### Decision 2: MVP 使用近似攀爬平面的空间哈希索引，最终距离按 3D 校验

推荐 MVP 使用固定网格空间哈希索引：初始化时把抓点世界坐标投影到“近似攀爬平面”的局部二维坐标，按 cell size 分桶；查询时把输入点投影到同一平面，只检查目标格子和相邻格子的抓点，再用抓点与查询点的 3D 世界坐标做精确距离、半径和稳定值校验。当前 Gym 看起来接近 2D 平面，但最终落地可能是有起伏的 3D 岩壁；因此平面 cell 只作为加速结构，不作为最终几何事实。

近似攀爬平面可以来自关卡根节点的 local X/Y、一个显式配置的 wall plane，或由 Gym 墙面参考 Transform 定义。抓点运行时数据 MUST 保留 3D 世界坐标，查询结果也 MUST 返回 3D 坐标。若岩壁起伏导致投影误差较大，查询应扩大邻近 cell 搜索范围或增大 cell size，而不是直接改用全量扫描。

理由：抓点总体沿主岩壁分布，平面分桶能保留简单、可调、低成本的查询加速；最终 3D 校验又能兼容局部凹凸、岩石起伏和抓点随地形摆放的情况。它比 KD-Tree 更容易增量调试，也比全量遍历更稳定。

备选方案：

- 全量遍历：适合几十个点以内的极早期原型，但不能作为高频接口契约。
- KD-Tree/Octree：查询效率更高，也更适合强 3D 分布，但实现和调试成本更高；只有当岩壁不再能视为近似平面，或平面分桶候选过多时再考虑。
- Unity Physics Overlap：可复用 Collider，但依赖物理层配置，且查询结果需要额外排序和过滤；MVP 可作为编辑器调试辅助，不作为核心数据源。

### Decision 3: 稳定值采用 `0-10` 的关卡事实值，并允许距离衰减

每个真实 `Anchor Point` 手动配置一个基础稳定值 `baseStability`，范围为 `1-10`。关卡系统对外返回的稳定值范围为 `0-10`：

- 查询点不在任何抓点有效半径内，返回 `0`。
- 最近或最合适的真实抓点在有效半径内时，返回该点基础稳定值经过距离衰减后的结果。
- 距离越接近抓点中心，返回值越接近 `baseStability`；越接近边缘，值越低，低到最小可抓阈值以下时由受力系统视为低质量或无效。

推荐公式：`stability = round(baseStability * falloff(distance / grabRadius))`，其中 `falloff(0)=1`，`falloff(1)=0`，默认可先使用线性衰减，后续按手感改为曲线配置。

理由：用户手部不是精确点选，稳定值需要同时表达“这个抓点本身质量”和“手当前摸得准不准”。`0-10` 比 `0-1` 更直观，方便关卡设计师调参和调试显示。

备选方案：

- 只返回抓点基础稳定值，不做距离衰减：实现更简单，但摸索边缘和中心的手感差异需要另一个系统处理。
- 只由受力系统计算稳定值：会让受力系统反向依赖关卡几何，破坏职责边界。
- 使用连续 `0-1`：数学上更自然，但当前需求明确最大为 `10`，MVP 应直接暴露 `0-10`。

### Decision 4: 最近抓点查询返回结构化结果而非裸坐标

最近抓点接口推荐返回 `AnchorPointQueryResult`，包含是否命中、抓点 ID、世界坐标、距离、基础稳定值、当前查询点稳定值、抓取半径和调试名。无可用抓点时返回 `found=false`，不抛异常。

理由：角色 3C、震动反馈、调试面板和受力适配器需要的信息不同。结构化结果可以避免每个调用方拿到坐标后再二次查表，也便于后续加入假点、障碍点或难度标签。

备选方案：只返回 `Transform` 或 `Vector3`。该方案短期方便，但调用方很快会需要稳定值和 ID，导致重复查询或直接依赖场景组件。

### Decision 5: 数据源由场景组件负责，运行时服务负责查询

`Anchor Point` 组件负责关卡作者配置：基础稳定值、抓取半径和启用状态。抓点 ID 在运行时收集后自动分配，不需要手动配置；调试名直接使用 `gameObject.name`；抓取半径直接使用现有 `previewSlightRadius`，让手感探测范围和实际抓取逻辑保持一致。`LevelAnchorRegistry` 或同等 Gameplay 模块负责初始化收集、校验数据、构建快照和索引。`ILevelAnchorQuery` 或同等接口只暴露运行时查询。

现有 `AnchorPoint` 脚本中的 `previewIntenseRadius` 和 `previewSlightRadius` 属于预研阶段 Gizmo 预览字段，分别对应 `MouseFollowJitter` 的剧烈抖动半径和轻微抖动/最远探测半径。正式实现中抓取半径直接使用 `previewSlightRadius`，不再新增 `grabRadius` 或迁移字段，避免手感探测和实际逻辑出现两套半径。

表面法线不作为关卡抓点 MVP 的必需数据。它主要服务于手部贴墙朝向、IK/吸附姿态和调试箭头，而这些信息更接近角色 3C 当前墙面 Raycast/Collider 命中结果，不应强制挂在每个抓点查询结果上。若后续某个系统确实需要抓点附近的表面方向，可在角色 3C 或墙面检测模块中提供，不进入本次基础抓点查询契约。

理由：场景组件贴近关卡编辑；运行时服务贴近玩法高频调用。两者分离后，后续可以替换数据来源，例如 ScriptableObject 烘焙数据或网络同步数据，而不影响调用方接口。

备选方案：把查询逻辑直接写在每个 `Anchor Point` 组件上。该方案组件自治性强，但很难高效回答“最近点”和“全局索引”问题。

### Decision 6: 初始化校验失败采用可视化警告，不阻塞 MVP 运行

初始化时应校验基础稳定值范围、`previewSlightRadius` 是否大于 0、运行时 ID 是否分配成功、坐标是否可被索引。MVP 阶段对异常数据优先记录警告并跳过无效点；在调试模式中显示被跳过的点位和原因。

理由：game jam 期间关卡会频繁改动，强制中断运行会拖慢调试。跳过无效点并给出明确日志能让开发继续，同时暴露数据问题。

备选方案：任何配置错误都抛异常停止 Gameplay。该方案更严格，适合正式生产质量门禁，但不适合当前快速迭代阶段。

## Risks / Trade-offs

- [Risk] 空间哈希 cell size 配置不合适导致查询漏点或候选过多 → Mitigation：默认 cell size 使用最大抓取半径或其倍数，并在调试面板显示查询格子和候选数量。
- [Risk] 距离衰减后的稳定值手感不符合预期 → Mitigation：把衰减曲线做成参数；早期先线性，后续根据试玩反馈调曲线或关闭衰减。
- [Risk] Gym 墙面存在 3D 起伏，平面投影索引可能漏掉跨 cell 的真实近邻 → Mitigation：索引只用于缩小候选范围，最终始终用 3D 世界距离校验；cell size 和邻域扩张需要覆盖最大抓取半径与预期地形起伏；复杂场景再切换三维空间哈希。
- [Risk] 关卡系统和受力系统都解释稳定阈值，造成规则重复 → Mitigation：关卡系统只返回 `0-10` 稳定事实；最小抓附阈值、稳定阈值和脱力判定仍由受力系统参数决定。
- [Risk] 运行时动态新增或禁用抓点需求出现 → Mitigation：MVP 先支持初始化静态快照；后续如有机关或破坏需求，再增加局部重建索引接口。

## Migration Plan

1. 在 OpenSpec 中新增 `level-anchor-system` 能力规格。
2. 实现阶段先定义 `Anchor Point` 数据字段、运行时数据结构和查询接口，不接入角色 3C。
3. 在 Gym 或测试场景中添加少量真实抓点，验证初始化收集、校验日志和最近点查询。
4. 构建空间哈希索引并用编辑器测试覆盖稳定值查询、最近点查询、空结果和边界半径。
5. 将受力适配器或角色 3C 改为通过关卡查询接口读取抓点事实。
6. 根据试玩反馈调整稳定值衰减、抓取半径和调试可视化。

Rollback 策略：若空间哈希索引实现影响进度，可保留同一查询接口并临时切换为带数量警告的全量遍历实现，调用方无需修改；后续再恢复索引实现。

## Open Questions

- 稳定值是否应只由 `Anchor Point` 基础值决定，还是采用“基础值 + 距离衰减”的推荐模型？
- 近似攀爬平面由哪个 Transform 或配置定义？推荐由关卡根节点或显式 wall plane 定义，而不是硬编码世界轴。
- 最近抓点接口是否需要限制最大搜索距离？推荐提供可选 `maxDistance`，用于避免远处抓点产生误导震动。
- 运行时是否存在会新增、移除或失效的真实抓点？如果没有，MVP 可以保持静态快照；如果有，需要在本能力中加入索引重建或增量更新要求。
