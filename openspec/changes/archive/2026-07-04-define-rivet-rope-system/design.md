## Context

Anchor 的核心循环是两名登山者用一根固定长度绳索和有限铆钉轮流领攀。现有策划文档已经把系统拆为角色 3C、关卡系统、受力系统、铆钉和绳索系统：受力系统只负责判断“是否掉落”，关卡抓点系统只提供抓点事实，铆钉和绳索系统负责“掉落后是否被保护、保护距离和伤害”。

项目处于 game jam 范围，Tuanjie/Unity 实现需要优先保证可玩、可调试和职责清晰。铆钉和绳索应先作为玩法规则系统落地，而不是追求真实绳索物理模拟。

## Goals / Non-Goals

**Goals:**

- 定义玩法铆钉的携带、部署、回收和总量约束。
- 定义固定长度绳索如何穿过已部署铆钉并约束两名玩家的距离。
- 在受力系统触发坠落后，根据玩家位置、绳长和已部署保护锚点输出保护结果、坠落距离和伤害输入。
- 支持下方玩家在上方玩家坠落时快速点击收绳，通过主动救援降低坠落伤害。
- 支持领攀/跟攀交替：前方玩家部署铆钉，后方玩家经过后回收铆钉，铆钉不在玩家之间远程互换。
- 提供参数配置和调试输出，便于在 jam 阶段快速调绳长、伤害曲线和交互范围。

**Non-Goals:**

- 不实现完整绳索网格、软体绳、连续碰撞或真实张力积分。
- 不负责左右手输入、IK、镜头、触觉反馈、角色动画或布娃娃表现。
- 不判定抓点稳定、耐力脱力或假点/障碍失效，这些仍由受力系统和关卡系统负责。
- 不直接扣除生命值；系统只输出结构化伤害结果，由生命系统消费。
- 不要求服务器权威判定；MVP 继续以本地玩法状态为准。

## Decisions

### Decision 1: 区分玩法铆钉和关卡 `AnchorPoint`

玩法铆钉使用独立运行时状态表示，例如 `RivetState`、`PlacedRivet` 或等价结构，包含铆钉 ID、所属玩家、部署位置、是否有效、是否在绳索路径中、调试名等。关卡 `AnchorPoint` 继续表示可抓取点或撒点生成的抓点事实，不等同于玩家部署的保护铆钉。

理由：当前项目里 `AnchorPoint` 已经被用于可抓点、震动半径和稳定值查询。如果把玩法铆钉复用为同一概念，会让“抓住哪里”和“保护在哪里”混在一起，后续调参和验收都容易混乱。

备选方案：复用 `AnchorPoint` 作为玩法铆钉。该方案短期省数据结构，但会让可抓点、可插锚点、已部署铆钉和保护锚点共享生命周期，不适合双人协作回收规则。

### Decision 2: 固定总长绳索使用逻辑路径和松弛量表现

MVP 中绳索总长默认使用背景设定中的 20 米，并通过配置允许调整。绳索不会因为打铆钉而变长；每次成功插锚只是把绳子新增一个“穿过点/转折点”。系统用两名玩家端点加上按攀爬进度排序的已部署铆钉形成逻辑路径，计算当前已占用路径长度、剩余松弛量和是否拉直，并输出 `Slack`、`Taut`、`ProtectedFall`、`UnprotectedFall` 等离散状态。

视觉上可用 `LineRenderer` 或等价组件表现绳索：两端应绑定到两名角色的腰部挂点；主玩法中表现挂点可以相对腰部向墙面方向和角色侧边做少量偏移，避免绳索从角色正前方穿插身体；拉直时路径段尽量接近直线并强化张力反馈；有多余长度时，路径段应拆成多个渲染点，叠加端点平滑、横向弧线、轻微下垂和小幅摆动，避免绳索像硬杆一样僵直。剩余松弛量还可以表现为下方玩家腰间/背包附近的一小圈绳环或 UI 松弛量提示。下方玩家主动收绳时，绳环缩小、松弛量减少，救援窗口结束后再按当前绳路状态恢复。视觉绳索不作为玩法判定源。

理由：规则化路径能稳定测试、方便调数，并且足够支撑“绳子拉直保护玩家”和“下方玩家收绳”的核心体验。把多余长度表现为绳环或松弛提示，比模拟整根软绳更适合 game jam。

备选方案：用 Rigidbody、Joint 和绳索段模拟真实张力。该方案表现更真实，但调参难、复现难，也会把保护规则绑定到物理误差上。

### Decision 3: 铆钉库存按玩家携带和场上部署守恒

系统维护总铆钉数，MVP 默认 5 个，但必须通过配置修改。每个铆钉在任意时刻只处于“某个玩家携带”或“场上已部署”两类核心状态之一。玩家只能部署自己携带的铆钉；任意玩家靠近场上已部署铆钉且自身处于 stable 状态时，交互层显示回收按钮，点击即可直接回收。回收后该铆钉进入点击回收者的携带库存，不允许直接把库存远程转给另一名玩家。

理由：该模型直接表达“前方玩家下锚、后方玩家回收、双方交替领头”的玩法。总量守恒也能避免 UI 显示、关卡状态和实际保护点不一致。

备选方案：只维护一个团队共享库存。该方案实现更简单，但会削弱“无法互换铆钉”和交替领攀的合作压力。

### Decision 4: 回收已部署铆钉会立即重算绳路

系统不设置“锁定/失效”这类阻止玩家操作的铆钉生命周期状态。若玩家在 stable 状态下靠近并点击回收一个正在参与绳路的铆钉，该铆钉立即从场上移除并进入回收者库存，绳索路径立即用剩余已部署铆钉重算。若被回收铆钉正好是队友坠落时的最高保护点，则坠落保护点可能下降或消失，估算自由坠落距离和伤害会立即按新绳路变大。UI 可以给出危险提示，但规则层不禁止回收。

理由：这符合“靠近且 stable 即可回收”的直觉，也避免引入玩家看不见的锁定状态。承重只是一帧绳路计算结果，不是铆钉生命周期状态。

备选方案：承重铆钉不可回收。该方案更安全，但会出现按钮规则和玩家直觉不一致，也会让“为什么不能拔”变成额外教学负担。

### Decision 5: 坠落保护基于最高已部署铆钉和剩余绳长

受力系统触发某名玩家坠落后，铆钉和绳索系统选择当前绳索路径中对坠落者有效的最高已部署铆钉作为保护点。若没有保护点，则输出无保护坠落。若有保护点，则根据 20 米默认绳长、保护点到下方玩家的已占用路径长度、保护点到坠落者的当前路径长度计算剩余松弛量和可下坠距离。MVP 可先用沿攀爬轴的近似距离，后续再替换为三维折线路径长度。

理由：策划期望“玩家跌落距离 = 绳子总长度 - 最上方锚钉与下方玩家之间的距离”的直觉规则。把它表达为可配置的路径长度计算，可以兼容近似实现和未来更精细的三维路径。

备选方案：按两名玩家直线距离计算保护。该方案更简单，但无法体现铆钉改变保护点的价值，也不能鼓励合理下锚。

### Decision 6: 坠落窗口允许下方玩家主动收绳减伤

当上方玩家触发坠落后，系统开启一个短暂救援窗口。下方玩家如果处于 stable、未坠落、未受击且可交互状态，可以快速点击“收绳”输入。每次有效点击会积累收绳进度或等价的收绳长度，优先减少剩余松弛量，结算时从估算自由坠落距离中扣除，或转化为额外减伤倍率。收绳效果必须有上限，不能把坠落伤害降到配置的最小伤害以下。窗口结束、绳索拉直到极限、坠落者落定或下方玩家失稳时，收绳结算停止。

交互反馈需要清晰：坠落发生时下方玩家看到收绳按钮高亮或救援提示；有效点击时绳索更紧、按钮/计量条变化，并可触发轻微震动或音效；无效点击时给出冷却、失稳、距离不足或不可救援反馈。MVP 采用快速点击按钮而不是复杂节奏输入，后续可以扩展为长按蓄力或节奏判定。

理由：该机制让下方玩家在队友坠落时不只是被动“站在正确距离”，而是能主动承担保护职责。快速点击易解释、实现成本低，也能制造紧张的双人协作瞬间。

备选方案：坠落伤害完全由铆钉和绳长自动决定。该方案规则清晰，但下方玩家在队友失误时缺少即时参与感，合作感偏弱。

### Decision 7: 伤害以结构化结果输出

系统输出类似 `RopeFallResolution` 的结果，至少包含是否受保护、保护锚点 ID、绳索是否拉直、估算自由坠落距离、主动收绳量、救援减伤倍率、伤害倍率、建议伤害值、主要原因和调试字段。生命系统负责最终扣血，角色 3C 负责表现和恢复。

理由：铆钉和绳索系统关注保护规则和伤害输入，不应直接修改生命值或角色表现。结构化结果便于测试、调试 UI 和未来联网同步。

备选方案：在绳索系统里直接扣生命并切换角色状态。该方案减少适配代码，但会让生命、角色表现和绳索规则紧耦合。

### Decision 8: 插拔锚通过角色/关卡提供的可交互事实驱动

插锚输入由角色 3C、正式 UGUI 操作按钮或交互适配器触发，系统只消费“玩家位置、墙面命中点、是否允许插锚、当前携带库存”等事实。拔锚同理，系统只在玩家进入回收范围、玩家自身处于 stable、目标为已部署铆钉时改变库存。收绳输入也通过角色 3C 或 UI 适配器提交为救援窗口内的离散点击事件，铆钉和绳索系统只计算点击是否有效、积累了多少收绳量以及如何影响伤害。主玩法 UI 不是常驻调试面板：插锚、回收和收绳救援入口必须按当前可执行条件显示；库存、场上数量和收绳量等提示只在相关交互上下文中展示。

理由：该边界让墙面 Raycast、UI 按钮、触觉反馈和交互表现留在角色 3C 或交互层，铆钉系统只负责规则结果。

备选方案：铆钉系统直接读取输入和 Raycast。该方案接入快，但会重复角色 3C 的触点映射逻辑，并削弱可测试性。

### Decision 9: 打钉和回收使用事件同步

打钉和回收都会改变共享绳路、双方库存和远端表现，因此成功结果必须通过 `game.event` 同步。MVP 采用客户端本地规则判定成功后发送事件的方式：打钉成功发送 `rivet.place`，回收成功发送 `rivet.collect`。事件 payload 至少包含 `eventId`、`actorPlayerId`、`rivetId`、铆钉世界位置或被回收铆钉 ID、操作后的携带库存、绳路版本号或等价序号。远端收到后按 `eventId` 去重，并更新只读铆钉列表、库存显示和绳索路径。

失败尝试不需要同步，除非后续需要远端失败表现。服务器仍只做 `game.event` 基础校验和转发，不理解铆钉规则，也不做服务器权威判定。若远端先通过 `game.state` 看到状态变化、随后收到事件，应以事件去重和版本号避免重复部署或重复回收。

理由：打钉和回收是低频、离散、对双方状态有明确影响的动作，用事件同步比等待下一帧 `game.state` 更清晰，也便于调试日志和远端音效/视觉反馈。

备选方案：只靠周期性 `game.state` 同步铆钉列表。该方案实现更少，但远端表现延迟更明显，且回收导致绳路重算时不易解释发生了什么。

## Risks / Trade-offs

- [Risk] 规则化绳索缺少真实摆动感 → Mitigation：MVP 先保证判定可信，视觉绳索可用 LineRenderer、端点延迟和简单摆动补充表现，不参与判定。
- [Risk] 领攀/跟攀身份规则过重，影响 jam 进度 → Mitigation：实现阶段先支持当前领攀者部署、后方玩家回收和手动/触发式换领，复杂越位规则后置。
- [Risk] 路径长度近似与 3D 墙面弯曲不一致 → Mitigation：接口保留路径长度计算器，MVP 用攀爬轴或折线距离，后续可替换为更精确算法。
- [Risk] 伤害曲线不好调导致保护价值不明显 → Mitigation：通过 ScriptableObject 暴露绳长、保护减伤、最小/最大伤害、自由坠落距离曲线和调试输出。
- [Risk] 玩家误回收正在保护队友的铆钉导致伤害暴增 → Mitigation：回收按钮仍可点击，但 UI/音效需要明确提示“正在保护中/回收有风险”，点击后立即按新绳路重算保护和伤害。
- [Risk] 快速点击收绳变成单人手速测试，压过攀爬协作 → Mitigation：设置短救援窗口、每秒有效点击上限、最大减伤上限和最小伤害，保证收绳只能补救而不能完全抵消错误。
- [Risk] 下方玩家在自身不稳定时仍能救援导致规则违和 → Mitigation：收绳要求下方玩家处于 stable 且可交互状态；若下方玩家脱力、坠落或受击，救援窗口立即失效或暂停。

## Migration Plan

1. 在 OpenSpec 中新增 `rivet-rope-system` 能力规格。
2. 实现阶段先新增纯 C# 规则模型、参数配置和编辑器/单元测试，不改动现有角色控制。
3. 接入角色 3C 的插锚/拔锚/收绳输入和玩家位置，接入受力系统的坠落触发事实。
4. 接入 `game.event` 同步，将打钉成功和回收成功作为离散事件发送，并在远端更新铆钉、库存和绳路表现。
5. 在测试场景中用可视化线段、收绳提示和调试面板验证库存、部署、回收、事件同步、绳长拉直、主动救援和坠落伤害。
6. 合入 `MainLevel` 主玩法场景，用运行时绑定器连接本地和远端角色腰部挂点；主场景不安装调试输入控制器，正式交互通过 UGUI 按钮提交插锚、回收和收绳请求。
7. 若接入后手感不理想，优先调整参数、动态绳索表现和路径长度计算器，不回滚角色 3C、受力系统或关卡抓点边界。

## Implementation Slices

1. `Anchor.RivetRopeSystem` 运行时程序集：先实现 `RivetRopeModel`、`RivetRopeConfig`、库存、已部署铆钉、绳路计算、坠落结算、收绳窗口和事件数据生成。
2. EditMode 规则测试：用假玩家位置和手写请求覆盖库存守恒、插锚、回收、绳长、坠落保护、收绳减伤和事件去重，不依赖角色 3C、受力系统或真实网络。
3. 调试驱动：提供 `RivetRopeDebugDriver` 和 Gizmos/日志，允许在临时场景中用两个 Transform 代表玩家，观察绳索路径、松弛量和坠落结果。
4. 主玩法绑定：在 `MainLevel` 安装 `RivetRopeMainGameplayBinder`，运行时用 3C 躯干位置加腰部偏移更新本地腰部挂点；双人联机时若存在远端 `Remote Climber`，另一端绑定到远端腰部。
5. 正式 UI：在 `MainLevel` 安装 `RivetRopeMainGameplayUi`，提供插锚、回收铆钉和收绳救援按钮；按钮只在满足库存/稳定状态、回收范围或救援窗口等条件时显示，点击时提交请求，不监听键盘或触摸区域。
6. 适配层：在角色 3C、受力系统、生命系统和联网模块稳定后，只把它们的输入/输出接到规则模型，不把规则复制到适配脚本里。

## Open Questions

- 起始铆钉库存是全部由首位领攀者携带，还是两名玩家按关卡配置分配？
- 换领条件是后方玩家回收到指定铆钉后自动切换，还是由玩家在安全状态下主动触发？
- MVP 是否需要支持绳索绕多个铆钉的完整折线，还是先只按最高已部署锚点计算保护？
- 收绳输入在移动端最终采用单按钮快速点击，还是允许长按但按点击频率换算救援强度？
