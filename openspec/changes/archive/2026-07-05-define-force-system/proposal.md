## Why

当前玩法已将“受力系统”列为核心系统之一，但它与角色 3C、关卡事实、耐力、障碍、假点、绳索保护之间的职责边界还没有形成可实现的契约。现在需要先定义受力判定的输入、输出和验收标准，让后续角色控制、关卡数据和铆钉绳索系统可以并行开发且减少互相耦合。

## What Changes

- 新增受力系统能力规格，定义角色在攀爬过程中的合法受力姿态判定。
- 明确受力系统只输出姿态/脱力/坠落等判定结果，不直接处理角色输入、关卡生成、绳索伤害或最终生命扣除。
- 定义左右手接触、抓点类型、抓点稳定值、障碍、假点、耐力、身体状态等输入如何影响稳定、单手脱力、双手脱力和坠落状态。
- 定义关卡抓点查询、受力输入适配器和纯规则评估器之间的接口边界，保证其他系统未完成时也能先完成受力系统单元测试。
- 建立面向 game jam MVP 的可调参边界：允许先用轻量规则和调试可视化验证手感，后续再替换更复杂的物理解算。
- 非目标：不在本变更中实现完整布娃娃物理、不实现铆钉库存/绳长伤害、不设计完整关卡编辑工具、不改变现有联网中继协议。

## Capabilities

### New Capabilities

- `force-system`: 定义 Anchor 攀爬玩法中角色受力姿态判定、状态输出、系统边界和 MVP 验收要求。

### Modified Capabilities

- 无。

## Impact

- 影响后续 Unity/Tuanjie 玩法脚本设计，预计涉及 `Assets/Anchor/` 下角色控制、关卡数据、调试 UI 或测试场景。
- 受力系统需要消费角色 3C 的手部接触状态、关卡系统提供的攀附点/障碍/假点事实与抓点稳定值，以及耐力状态。
- 受力系统需要向角色 3C 输出稳定/脱力/坠落等状态，并向铆钉和绳索系统提供“已经坠落”的触发事实。
- MVP 阶段不引入新的第三方物理依赖，优先使用 Unity/Tuanjie 现有组件、ScriptableObject 配置和轻量 C# 规则实现。
