## Context

`rivet-rope-system` 现有规格已经定义了坠落保护、主动收绳和结构化伤害结果，并明确该系统不直接扣生命值。`docs/mao-gamejam.md` 中也已将生命定义为核心属性：每次掉落消耗生命值，伤害与掉落距离正相关。当前缺口是主玩法还没有独立的生命值归属、伤害应用和归零失败反馈。

本变更横跨规则计算、玩家状态和 UI 表现：铆钉绳索系统继续负责输出伤害建议值，新增生命失败能力负责消费该结果并决定是否失败。

## Goals / Non-Goals

**Goals:**
- 定义玩家生命值、当前生命、扣血和失败状态的最小交付边界。
- 明确坠落伤害以“坠落者到第一个保护铆钉的绳索段长度”为主要距离输入，且距离越长伤害不降低。
- 生命值归零后立即进入失败状态，并显示 MVP 失败弹窗。
- 保持伤害计算、生命扣减和 UI 表现之间的职责边界清晰，便于后续调参和联网同步扩展。

**Non-Goals:**
- 不实现复活、重试、关卡结算页、排行榜或复杂失败动画。
- 不要求 relay 服务器理解生命值或失败规则。
- 不把绳索视觉物理、材质表现或第三方 Rope Swing 状态机纳入伤害计算。
- 不改变现有耐力耗尽触发坠落的事实来源；生命系统只消费坠落伤害结果。

## Decisions

### 1. 使用独立生命失败能力消费伤害结果

生命值归属放在 `player-health-failure`，而不是 `rivet-rope-system`。铆钉绳索系统输出 `FallDamageResult` 或等价结构化结果，生命系统接收后扣除对应玩家生命值。

原因：现有规格已经要求绳索系统不直接修改生命值。保持这个边界可以让伤害曲线和生命 UI 独立调参，也避免未来障碍点、假点或其他伤害来源都依赖绳索模块。

备选方案：让绳索系统直接扣血。该方案实现更快，但会破坏职责边界，后续加入障碍伤害或联网状态采样时更难维护。

### 2. 将“第一个锚点”定义为坠落者沿当前绳路回溯遇到的第一个已部署保护铆钉

伤害距离使用坠落者端点到第一个保护铆钉之间的逻辑绳索段长度。若当前没有保护铆钉，则使用无保护坠落分支，并输出明确原因。

原因：该定义与现有绳索路径、最高保护铆钉和局部张力方向模型兼容，同时能直接表达“离第一个锚点越远，摔得越疼”的设计意图。

备选方案：使用世界垂直高度差或玩家到最高铆钉的直线距离。它们更直观，但会绕开现有绳路模型，且多铆钉转折时不符合绳索保护的玩法反馈。

### 3. 失败状态优先冻结主玩法交互

生命值扣至 0 或以下时，生命系统立即进入失败状态，UI 显示失败弹窗，并通知 3C、输入、绳索救援入口或主玩法控制器停止继续处理恢复攀爬、收绳和插拔锚等交互。

原因：用户需求是“血量为零直接失败。弹一个失败弹窗即可”。MVP 中失败反馈要稳定明确，先冻结交互比继续让角色恢复或可操作更符合预期。

备选方案：允许落定后再失败或保留短暂操作窗口。该方案表现更丰富，但会引入额外状态竞争，不适合当前范围。

### 4. MVP UI 只提供失败弹窗

失败弹窗只负责显示失败结果和阻止玩家误以为还能继续。按钮、重开流程、房间返回、联网失败同步可作为后续变更处理。

原因：当前需求只要求弹窗；保持 UI 简单能降低主玩法接入风险。

### 5. 伤害触发必须接入主玩法实际在用的坠落链路

实现前必须确认当前场景实际运行的坠落入口。`MainLevel` 当前由 `Climb3CLevelBinder` 运行时创建 `ClimbController3D`，联网模式也通过 `AnchorNetworkDemoController` 动态创建同一类 binder；`ClimbController3D` 在 `ForceEvaluator` 输出 `FallTriggered` 后调用 `BeginFall()`，并在 `UpdateFalling()` 达到落定条件后调用 `Land()`。本变更的伤害结算 MUST 接入这条在用链路：以 `BeginFall()` 或等价事件创建一次坠落上下文，以 `Land()`、救援窗口结束或等价落定事件触发一次性伤害结算。

`ForceSystemDebugDriver`、绳索视觉实验场景和单元测试中的 `ResolveFall(...)` 只能作为验证或规则计算入口，MUST NOT 作为 MainLevel 主玩法扣血触发源。

原因：项目中同时存在测试驱动、视觉实验和主玩法 3C 多条“坠落”相关路径。只接入实际在用的 `ClimbController3D` 可避免出现测试场景扣血正常但 MainLevel 不生效，或远端表现对象误触发本地扣血的问题。

## Risks / Trade-offs

- [Risk] “第一个锚点”在多铆钉路径中的玩家理解可能不一致 → Mitigation: 在规格中绑定为逻辑绳路上坠落者相邻的第一个保护铆钉，并在调试输出中显示该铆钉 ID 和绳索段长度。
- [Risk] 生命系统和耐力恢复流程同时响应坠落落定，可能出现失败后又恢复攀爬 → Mitigation: 失败状态必须优先于恢复逻辑，3C 在失败后不得重新启用输入和攀爬控制。
- [Risk] 当前网络 MVP 未定义生命值同步，双端失败表现可能暂时不一致 → Mitigation: 本变更先保证本地权威玩家失败；如需双端一致弹窗，后续通过 `game.state` 或 `game.event` 扩展协议。
- [Risk] 伤害曲线过陡会导致频繁秒杀 → Mitigation: 使用可配置最大生命、基础伤害、距离倍率、最小/最大伤害限制，并通过 EditMode 或手工验证调参。
- [Risk] 接入了未在 MainLevel 使用的测试坠落入口，导致实际玩法不扣血或重复扣血 → Mitigation: 实现任务必须先确认场景和联网模式中实际创建的 `ClimbController3D`，并以 `fallId` 保证一次坠落只结算一次。

## Migration Plan

1. 确认 MainLevel 单机和联网模式实际使用的 `ClimbController3D` 创建与坠落状态切换路径。
2. 在铆钉绳索伤害结果中补充或明确第一个保护铆钉 ID、到该铆钉的绳索段长度和调试原因。
3. 新增玩家生命状态组件或服务，初始化最大生命值和当前生命值。
4. 将在用坠落链路的落定/结算事件接入生命系统，扣血后广播生命变化或失败结果给 UI 和主玩法控制器。
5. 在主玩法 UI 中新增失败弹窗，失败时显示并禁用继续交互。
6. 补充规则单元测试和 MainLevel 冒烟测试。

回滚策略：保留绳索系统原有伤害结果输出，移除生命系统订阅和失败 UI 绑定即可恢复到仅输出建议伤害、不扣生命的行为。

## Open Questions

- 最大生命值和伤害曲线默认值需要在实现时根据当前关卡节奏初调；建议先使用易于手工验证的配置值。
- 双人联网时是否要求一名玩家失败后两端同时显示失败弹窗，建议作为后续联网协议变更明确。
