## Why

当前已经具备房间流程、MainLevel 主关卡、主玩家和基础攀爬能力，但房间进入游戏后缺少明确的玩家生成、身份绑定和远端表现同步规则。现在需要把“房主开始后双方怎样进入同一关卡、怎样看到并同步另一个玩家”定义清楚，避免后续客户端、玩法和协议配置各自实现一套不兼容逻辑。

## What Changes

- 明确房间进入 `room.inGame` 后，两个客户端进入同一个 MainLevel，并根据房间玩家列表为本地玩家和远端玩家分配稳定的 `host` / `guest` 槽位。
- 定义 MainLevel 开局生成规则：本地玩家使用可控制角色，远端玩家复用攀爬角色骨架做只读表现；房主固定作为先锋攀登者使用上方起攀主抓点，非房主固定作为第二攀登者使用下方起攀主抓点，进入后由主抓点和临近抓点推导合法初始位置。
- 扩展 `game.state` payload，用于同步玩家位置、朝向、攀爬状态、左右手抓点、耐力和必要调试字段。
- 扩展按需使用的 `game.event` payload，用于同步有明确远端表现、UI 或调试需求的一次性攀爬事件；插锚、拔锚和收绳事件先保留协议框架，随铆钉绳索系统实现时一起接入。
- 定义远端玩家表现策略：远端状态以对方客户端发送的状态为权威，本地只做插值、补帧和表现，不反向驱动对方玩法判定。
- 保持 relay 服务器职责不变：服务器继续只理解 `system.*` 与 `room.*` 控制流程，对 `game.*` 做基础校验并原样转发。

## Capabilities

### New Capabilities

无。本变更落在已有 `coop-networking` 联网能力内。

### Modified Capabilities

- `coop-networking`：补充 MainLevel 开局、房主先锋攀/非房主第二攀登者出生点绑定、本地与远端玩家对象职责、基础攀爬状态同步、按需一次性事件同步和断线表现要求。

## Impact

- 影响 `docs/coop-network-protocol.md` 与 `docs/coop-network-protocol.config.json` 中的 `game.state` / `game.event` payload 示例和说明。
- 影响 Unity/Tuanjie 客户端的房间进入游戏流程、MainLevel 初始化、玩家 prefab 生成、本地/远端玩家对象绑定、网络消息收发和调试显示。
- 不要求修改 relay 服务器的玩法 payload 解析；只有在发现 `room.inGame` 或房间玩家列表不足以支持槽位映射时，才需要补充房间控制字段。
- 不引入账号、昵称、匹配、重连、服务器权威物理或反作弊。

## Goals

- 两个客户端从同一房间进入 MainLevel 后，都能看到自己和对方的玩家角色。
- 每个客户端只控制自己的本地玩家，并通过 `game.state` 将基础攀爬表现同步给对方。
- 远端玩家使用只读攀爬角色骨架，能够跟随对方的攀爬位置、抓握状态和必要关键事件进行可理解的表现更新。
- 协议字段通过 JSON 配置维护，新增攀爬 payload 字段不要求重新部署服务器。

## Non-Goals

- 不实现服务器权威角色物理、抓点判定、绳索判定或伤害判定。
- 不实现断线重连后的完整状态恢复，只提供 MVP 级离开提示和远端冻结/隐藏策略。
- 不实现三人及以上、观战、匹配、角色选择或昵称系统。
- 不保证高精度竞技同步；MVP 目标是双人能看见彼此并完成基础协作验证。

## Acceptance

- 两个客户端完成连接、建房、加入、房主开始后，都会进入 MainLevel。
- `host` 与 `guest` 在两端映射一致，房主固定为上方先锋攀登者，非房主固定为下方第二攀登者；本地玩家可操作，远端玩家不可被本地输入控制。
- 本地玩家移动或基础攀爬时，另一端能看到远端玩家位置、朝向、攀爬状态和左右手抓握信息更新。
- 如当前实现接入了攀爬事件，触发事件后另一端能收到并在日志、UI 或远端表现中体现。
- 任一玩家离开房间时，另一端能停止更新该远端玩家并显示可调试状态。
