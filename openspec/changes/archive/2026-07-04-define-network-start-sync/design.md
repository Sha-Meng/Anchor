## Context

当前联网能力已经覆盖连接 relay、创建/加入房间、房主开始、切换到 demo 游戏场景、发送 `game.state` 和 `game.event`。现有实现位于 `Assets/Anchor/Networking/AnchorNetworkDemoController.cs`，目前游戏阶段仍使用 `NetworkDemoGame` 和两个 cube 验证转发；攀爬主关卡 `Assets/Scenes/MainLevel.scene` 已存在，基础攀爬通过 `Climb3CLevelBinder` 接入场景抓点和本地角色控制。

这次变更要把 demo 同步从“转发 cube 位置”推进到“房间进入 MainLevel 后生成自己和队友，并同步基础攀爬表现”。约束是：relay 服务器继续保持轻量，只处理 `system.*` / `room.*` 与 `game.*` 转发；攀爬、抓点、绳索和角色物理仍由客户端玩法系统负责。

## Goals / Non-Goals

**Goals:**

- 房主开始后，两个客户端进入 `MainLevel`，并基于同一份房间玩家列表得到一致的 `host` / `guest` 槽位。
- MainLevel 中生成一个本地可控玩家和一个远端只读表现玩家，两者都复用攀爬角色骨架；房主固定为先锋攀登者并使用上方起攀主抓点，非房主固定为第二攀登者并使用下方起攀主抓点，进入后由主抓点和临近抓点推导合法初始位置。
- 本地客户端以自己为权威发送基础攀爬状态，远端客户端只用这些状态驱动远端只读攀爬骨架表现对象。
- 网络层通过只读状态采样接口读取本地攀爬状态，不侵入主角输入、受力、耐力或状态机逻辑。
- 只有存在明确远端表现、UI 或调试需求的一次性动作才通过 `game.event` 传递；MVP 以整体状态基本同步为优先。
- `game.*` payload 字段继续由 JSON 协议配置和协议文档维护，不要求服务器部署变更。

**Non-Goals:**

- 不实现服务器权威物理、抓点判定、绳索判定、伤害判定或反作弊。
- 不实现断线重连后的完整世界恢复。
- 不实现多人扩展、观战、匹配、昵称、角色选择或账号系统。
- 不追求帧级一致的竞技同步；MVP 只要求双端能稳定看到彼此基础攀爬表现。

## Decisions

### 1. 游戏场景目标改为 MainLevel

房间收到 `room.starting` 后，客户端应加载 `MainLevel`，而不是继续加载 `NetworkDemoGame`。`NetworkDemoGame` 可作为旧的 relay 验证场景保留，但默认双人攀爬流程应指向主关卡。

备选方案是继续保留独立 demo 场景并复制 MainLevel 抓点。该方案能降低接入风险，但会让验收和真实玩法分离，后续还要再迁移一次，因此不作为主路线。

### 2. 使用房间玩家列表派生稳定槽位

`room.updated` 中的 `players` 列表和 `hostId` 是槽位来源。客户端按 `hostId` 标记 `host`，另一名玩家标记 `guest`。开局身份与起攀主抓点由房间身份决定，而不是由每个客户端本地的“1P/2P”视角决定：`host` 永远是先锋攀登者，使用配置中的上方起攀主抓点；`guest` 永远是第二攀登者，使用配置中的下方起攀主抓点。客户端进入 MainLevel 后直接找到对应主抓点和临近抓点，由这两个合法抓点分配左右手初始抓握并推导躯干位置。槽位只用于起攀点、颜色、UI 标识和本地/远端对象绑定，不作为长期身份。

备选方案是在客户端按加入先后自行推断槽位。该方案在晚收到消息、重建 UI 或重新进入场景时容易产生两端不一致，因此只作为缺少玩家列表时的临时 fallback。

### 3. MainLevel 会话对象负责生成和绑定

新增或扩展一个持久的网络会话对象，保存 `playerId`、`roomId`、`isHost`、房间玩家列表、当前槽位和消息序号。进入 MainLevel 后，由场景内 binder 或网络会话协调生成：

- 本地玩家：接入 `Climb3CLevelBinder` 或等价本地攀爬控制器，启用输入、相机、UI 和本地状态采样。
- 远端玩家：复用攀爬角色骨架作为只读表现对象，禁用本地输入、相机控制、受力判定、耐力消耗和本地玩法判定，只消费网络状态与必要事件。

备选方案是让 `AnchorNetworkDemoController` 继续直接创建所有运行时对象。短期改动少，但会让房间 UI、场景加载、攀爬绑定和远端表现耦合在一个脚本里；本次实现可以先小步拆出会话/绑定职责，再保留旧 UI 入口。

### 4. 客户端自权威同步模型

每个客户端只发送自己控制角色的状态，schema 使用 `climb-player-state.v1`。最小字段包括 `playerId`、`slot`、`position`、`rotationY`、`movementState`、`leftHandGripId`、`rightHandGripId`、`stamina`、`isFalling` 和调试用 `seq` / `sentAt`。网络层通过 `IClimbStateSource` / `ClimbStateSnapshot` 风格的只读采样接口读取本地攀爬状态，不直接访问或修改主角控制内部流程。远端收到状态后，按 `senderId` 找到远端对象，丢弃来自自己的状态和旧序号状态，并用小缓冲插值更新表现。

备选方案是同步输入而不是状态。输入同步更接近确定性模拟，但当前攀爬涉及触摸、IK、布娃娃、震动、抓点查询和后续绳索系统，MVP 阶段难以保证双端确定性，因此先采用状态同步。

### 5. 边沿动作使用 game.event

`game.state` 适合持续姿态，是 MVP 同步主路径。`game.event` 只负责有明确远端表现、UI 或调试需求的一次性动作；没有表现需求、且能通过下一帧状态自然反映的动作不需要发送事件。schema 使用 `climb-event.v1`，事件必须携带 `eventId`、`eventType`、`actorPlayerId` 和可选 `data`，远端用 `eventId` 去重。首批事件按实现需要接入，优先考虑 `fall.start` / `fall.recover` 等状态快照难以表达边沿反馈的动作。插锚、拔锚和收绳是协作玩法需要的事件类型，但本变更只保留协议和分发框架，具体发送时机、payload 和远端表现随铆钉绳索系统实现时一起确定。

备选方案是把所有内容塞进状态快照。这样实现简单，但状态发送频率低或丢包时容易错过瞬时反馈，玩家会看不到队友“刚抓住/刚松手/刚坠落”的关键协作信号。

### 6. 断线按 MVP 方式处理

收到 `room.peerLeft` 后，本地停止更新对应远端玩家，将其冻结、置灰、隐藏或显示“队友离开”调试状态，并允许本地返回房间入口。MVP 不恢复远端最后完整状态，也不尝试重新匹配。

备选方案是立即销毁远端对象。销毁简单，但会丢失调试现场；冻结或置灰更利于测试同步中断点。

## Risks / Trade-offs

- [Risk] `room.updated` 当前客户端解析只保存 `playerCount`，没有保存完整 `players`。→ 在实现时补齐玩家列表解析；若 relay 未返回 `players`，先根据 `hostId` 与自身 `playerId` 做临时槽位推断，并在协议文档标明限制。
- [Risk] `Climb3CLevelBinder` 当前默认只生成一个本地攀爬角色。→ 拆出可复用的本地/远端角色生成路径；本地启用控制器，远端复用攀爬角色骨架但只读应用网络状态。
- [Risk] 直接同步 Transform 可能与后续绳索/物理表现不一致。→ MVP 只把远端对象当表现层；涉及伤害、受力、绳索保护的最终判定继续由本地权威玩家产生并通过事件/状态展示。
- [Risk] 移动网络抖动会导致远端抖动或延迟明显。→ 使用 10Hz 左右发送频率、序号丢旧包和 100-200ms 插值缓冲；先优化可读性，不追求低延迟竞技手感。
- [Risk] `MainLevel` 场景可能缺少网络入口 bootstrap。→ 在场景或运行时自动确保网络会话对象存在；手工从 MainLevel 单机启动时允许降级为本地单人攀爬。

## Migration Plan

1. 保留现有房间入口和 relay 连接流程，先把游戏场景常量从 `NetworkDemoGame` 切到 `MainLevel`，或改为可配置目标场景。
2. 在客户端保存房间玩家列表和槽位信息，进入 MainLevel 后初始化本地/远端玩家绑定。
3. 扩展协议配置和协议文档，新增 `climb-player-state.v1` 与 `climb-event.v1` payload 字段说明。
4. 将现有 demo cube 状态发送替换为攀爬状态采样接口；远端使用只读攀爬角色骨架验证位置、手部抓点和状态更新。
5. 增加按需事件发送与去重处理，只接入当前远端表现确实需要的一次性事件。
6. 完成双客户端 MainLevel 冒烟验收后，再逐步提升远端骨架插值质量。

回滚策略：保留旧 `NetworkDemoGame` 或配置项，若 MainLevel 接入失败，可临时切回 demo 场景验证 relay 和房间流程。

## Open Questions

无。当前方案已收敛为：远端复用攀爬角色骨架做只读表现；房主固定使用上方先锋攀登者起攀主抓点，非房主固定使用下方第二攀登者起攀主抓点，并由主抓点和临近抓点推导合法初始位置；网络层通过只读状态采样接口读取本地状态；事件同步按表现需要最小接入，插锚/拔锚/收绳随铆钉绳索系统一起落地。
