## Context

现有 `rivet-rope-system` 已经负责铆钉库存、插拔锚、绳索路径、绳长约束、坠落保护和联网同步事实。当前表现层主要通过 `RivetRopeLineVisual` 使用 `LineRenderer` 将逻辑路径绘制为多段曲线，并带有简化的下垂、摆动和约束平滑，但它仍然是渲染点模拟：没有真实绳段碰撞、没有可辨识绳索材质，也不能给玩家明确传达“松弛、受力、拉紧、回弹”的物理反馈。

项目内 FImpossible Creations 的 Rope Swing demo 展示了 ragdoll 角色通过挂点进入摆荡状态、由身体刚体速度驱动移动器、并用 `LineRenderer` 显示连接线的方式。它适合作为角色受力、挂点和摆荡反馈参考，但 demo 目标是单角色 rope swing，不等同于 Anchor 的双人铆钉绳路、绳长守恒、回收铆钉和联网同步规则。

## Goals / Non-Goals

**Goals:**

- 让绳索在松弛、拉紧、端点移动、铆钉转折和坠落保护时具备更可信的动态表现。
- 让绳索材质更像实体绳：有宽度、纹理/颜色、表面细节、端点连接和状态反馈。
- 保持表现模拟与玩法规则解耦，物理和材质效果不得改变绳长、库存、保护点、伤害和同步事实。
- 设计一个专用绳索视觉测试场景，能直观看到绳索材质、端点跟随、松弛/拉紧、铆钉转折和表现模式切换。
- 支持在测试场景中快速切换或调参，对比默认轻量表现和更强物理表现。
- 评估 FImpossible Rope Swing / ragdoll 示例可借鉴的挂点、摆荡和角色受力反馈模式。

**Non-Goals:**

- 不重写铆钉绳索核心规则模型。
- 不把第三方 demo 代码直接变成 `rivet-rope-system` 的规则依赖。
- 不在本变更中完成最终 3C 拉拽、手部 IK、镜头或生命值扣除逻辑。
- 不要求联网同步每个绳段物理粒子；远端应从同一逻辑绳路本地重建表现。

## Decisions

### 决策 1：默认采用增强版程序化绳索表现

默认实现继续基于逻辑绳路和 `LineRenderer`，但将当前散落在视觉脚本中的参数提升为明确的视觉配置：绳宽、材质、UV 平铺、松弛下垂、端点跟随、阻尼、约束迭代、摆动幅度、拉紧颜色/张力反馈等。这样能以较低性能成本覆盖主玩法需要，并且与现有 `RopePathResult`、验证场景和联网规则最兼容。

替代方案是立即改为完全刚体/关节链绳索。它能提供真实碰撞和受力，但调试成本、性能风险、联网差异和角色控制耦合都会更高，不适合作为默认路径。

### 决策 2：新增可选物理绳段模式，而非替换逻辑绳路

物理绳段模式用于更高表现需求：在两名玩家和铆钉之间生成一组轻量绳段点或刚体节点，通过 Verlet/位置约束或 Joint 链跟随逻辑端点。它只服务表现层，可读取 `RopePathResult` 和端点 Transform，不向核心模型写回绳长或保护结果。

优先评估 Verlet/位置约束方案，其理由是可控、轻量、容易固定首尾和铆钉转折点，并能在 `LateUpdate` 或固定步长中稳定重建。Joint 链方案保留为实验选项，适合需要真实碰撞或与 ragdoll 交互的局部演示，但不作为 MVP 默认。

### 决策 3：FImpossible 示例作为交互反馈参考

FImpossible Rope Swing demo 可参考三类做法：命中挂点后切换角色状态、挂点驱动 ragdoll/移动器速度、玩家输入向身体链条施加摆动冲量。Anchor 可以借鉴这些反馈来设计未来的坠落/摆荡表现，但本变更只要求绳索表现侧保留对 ragdoll 挂点或角色受力组件的适配边界。

不直接复用 demo 的原因是它依赖具体角色控制器、ragdoll 配置和单点摆荡假设；直接接入会污染双人铆钉绳索的规则边界。

### 决策 4：材质表现通过资源和安装器进入场景

绳索应有专用材质或材质预设，支持沿绳长方向平铺纹理，并根据松弛/拉紧状态调整颜色、宽度或高亮强度。主玩法安装器和验证场景构建器应能自动配置默认材质，避免手工搭建时退化成 Unity 默认线条。

### 决策 5：验收以视觉行为和规则隔离为核心

验收不要求像物理引擎那样逐帧一致，而是验证玩家可感知结果：松弛时下垂和轻微摆动，拉紧时变直并有张力反馈，端点移动时有阻尼跟随，材质不像纯色细线。同时需要验证表现调整不会改变已有 EditMode 规则测试结果。

### 决策 6：新增专用 `RivetRopeVisualLab` 测试场景

当前验证场景更偏规则烟测，不适合观察绳索本身。新增或重建一个专用绳索视觉测试场景，建议命名为 `RivetRopeVisualLab.scene`，定位为“绳索表现观察台”，而不是完整玩法关卡。

场景布局采用固定侧视或轻微俯视镜头，背景保持简洁，放置两个可移动玩家端点、三到五个可开关的铆钉点、清晰的距离刻度和状态文本。端点需要支持自动往返移动、手动拖动或按钮驱动移动，方便观察绳索跟随、回弹和拉紧过程。铆钉点应提供直连、单转折、多转折和近距离急转弯几种预设路径，方便比较材质平铺和转折可读性。

测试场景的 UI 应提供少量关键控制：表现模式、绳长、松弛量/下垂强度、阻尼、摆动、宽度、材质/颜色、铆钉预设和自动运动开关。界面同时显示逻辑路径长度、剩余松弛量、张力状态、渲染点数量和当前模式，避免开发者只能凭肉眼猜测。

该场景不要求接入完整双人 3C、联网或真实关卡输入，但端点对象应模拟腰部挂点语义，确保主玩法接入时观察结果有参考价值。

## Risks / Trade-offs

- [风险] 程序化表现仍不是真实绳索碰撞，可能在复杂地形穿插。→ 缓解：默认接受为 MVP 表现；物理绳段模式用于后续局部碰撞需求。
- [风险] 物理绳段在远端客户端表现不完全一致。→ 缓解：不同步绳段粒子，只同步铆钉事实；远端本地重建，规则结果以核心模型为准。
- [风险] 绳索材质资源在 Tuanjie/Unity 渲染管线中表现差异较大。→ 缓解：先使用内置管线兼容材质和简单纹理，提供可替换字段。
- [风险] FImpossible demo 的 ragdoll 依赖较重。→ 缓解：只把它作为参考和可选适配，不纳入默认 asmdef 依赖。
- [风险] 过多调参项会增加调试复杂度。→ 缓解：提供少量预设和 ScriptableObject 默认值，测试场景只暴露关键参数。
- [风险] 测试场景过于抽象，无法暴露主玩法中的跟随问题。→ 缓解：端点使用腰部挂点语义和自动运动路径，并提供直连、单锚、多锚和急转弯预设。

## Migration Plan

1. 在现有 `RivetRopeConfig` 或独立视觉配置中增加绳索表现参数和材质引用。
2. 增强 `RivetRopeLineVisual` 或拆出 `RivetRopeVisualController`，保留现有逻辑路径输入。
3. 更新主玩法安装器，为绳索配置默认材质、宽度和表现参数。
4. 新增或重建 `RivetRopeVisualLab` 测试场景，提供端点移动、铆钉路径预设、参数面板和状态文本。
5. 增加可选物理绳段组件，默认关闭或仅在测试场景可切换。
6. 扩展 EditMode/测试场景验收，确保表现配置不改变核心规则输出。

回滚策略：保留原 `LineRenderer` 直连路径作为最小 fallback；如物理绳段模式不稳定，可关闭该模式但保留材质和基础动态表现。

## Open Questions

- 最终游戏视角下绳索是否需要与墙面/角色发生可见碰撞，还是只需要避免最明显穿模？
- 绳索材质是否已有美术资源可用，还是需要先创建程序化/占位纹理？
- 未来坠落时是否希望角色被绳索实际拉拽摆荡，还是先只做视觉张力反馈并把运动交给 3C？
