## Context

当前 `rivet-rope-system` 已经维护铆钉库存、逻辑绳路、绳长状态、坠落保护和视觉绳索要求。现有规格也明确了绳索系统输出绳长约束结果，但不得直接移动角色；角色 3C 负责移动、IK、相机、耐力和布娃娃切换。

这次变更需要补上两者之间的中间层：绳索系统计算可测试、确定性的“力反馈结果”，角色 3C 再把它转换成运动约束和玩家反馈。这样可以让绳索从纯表现升级为真实玩法反馈，同时不破坏现有系统边界。

主要约束：

- Unity/Tuanjie 项目内实现，不引入新外部依赖。
- 逻辑计算必须可在 EditMode 测试中脱离场景、Collider、Rigidbody 和视觉绳段运行。
- 联网同步继续同步铆钉事件、玩家端点和必要玩法事实，不同步每帧绳段粒子或 Joint 状态。
- 视觉绳索保持当前实现和美术表现，不以本变更重做绳子外观；现有程序化/Verlet 表现只作为显示层，不参与角色反馈规则。
- 可参考 FImpossible Creations 的 Ragdoll Animator Rope Swing demo 中“磁吸/拖拽刚体骨骼产生自然身体反馈”的思路，但不直接复用其整套绳索视觉或 demo 控制流。

## Goals / Non-Goals

**Goals:**

- 输出结构化绳索力反馈结果，包括是否激活、张力方向、张力强度、约束距离、建议速度修正、端点约束点和调试原因。
- 让角色 3C 在攀爬非布娃娃阶段消费该结果，形成可感知的限距、减速、拖拽、轻微回弹和提示，并尽量借鉴 FImpossible demo 中“对身体施加柔性牵引”的手感。
- 支持铆钉转折路径：张力方向应来自当前受约束端点沿逻辑绳路回到最近有效约束点的方向，而不是简单使用两名玩家直连方向。
- 保持计算确定性和参数可调，支持测试场景快速调试手感。
- 优先在现有绳索测试关卡中验证角色反馈手感，再决定是否接入主玩法。

**Non-Goals:**

- 不实现完整绳段碰撞或整根绳子的刚体链物理。
- 不重做当前绳索视觉、材质、LineRenderer/Verlet 显示或测试关卡的外观表现。
- 不让绳索系统直接修改角色 Transform、手部 IK、镜头或生命值。
- 不改变插锚、回收、库存守恒、坠落保护和伤害计算的既有规则。
- 不要求服务器理解绳索物理，也不同步视觉绳段的逐帧内部状态。
- 不在 MVP 阶段直接接入 FImpossible Rope Swing demo 的完整挂钩、发射、切换 fall state 流程。

## Decisions

### 1. 使用“反馈结果”而不是直接施力

绳索系统新增类似 `RopeForceFeedbackResult` 的数据结构，输入为逻辑绳路结果、被约束玩家端点、端点速度、deltaTime 和配置。输出只描述“应该产生什么约束/张力”，不执行位移。

选择原因：现有边界已经要求绳索系统不移动角色。结构化输出能被 3C、调试 UI、测试和联网回放共同消费，也便于在 EditMode 中验证。

替代方案：在绳索系统内部直接修改角色位置。该方案落地快，但会让规则层依赖角色实现，破坏测试边界，也容易和 3C 的 IK、胶囊体防穿模、布娃娃切换冲突。

### 2. MVP 采用“限距 + 阻尼 + 轻回弹”的确定性模型

当逻辑路径长度未达到绳长阈值时，反馈为空或低强度预拉紧提示；当路径长度超过绳长时，计算超出长度作为约束距离，并根据超出长度、端点远离速度和配置曲线得到张力强度。建议速度修正只抵消沿张力反方向继续远离的速度分量，并可附加有限的回弹。

选择原因：该模型足够表现“绳子拉住人”，实现成本低，可调参数少，且不需要依赖 Unity 物理步进稳定性。

替代方案：直接使用 SpringJoint/ConfigurableJoint。该方案更物理，但对角色驱动、布娃娃混合、联网确定性和移动端稳定性风险更高，适合作为后续可选表现/实验模式。

### 2a. 借鉴 FImpossible 的身体反馈思路，但只抽象为 3C 反馈策略

FImpossible Rope Swing demo 的自然感主要来自 `RA2MagnetPoint` 持续把某个 ragdoll 刚体骨骼拖向磁吸点，再由 Ragdoll Animator 的肌肉、关节和物理混合产生摆动。当前变更只借鉴这个“柔性牵引身体”的反馈思路：把绳索张力转换为躯干/角色目标的柔性拉回、速度阻尼和轻微回弹，而不是引入其视觉绳、发射挂点或完整 ragdoll 控制流程。

选择原因：这样可以获得更自然的被拉住感觉，同时保持 Anchor 自己的逻辑绳路、铆钉规则、测试场景和联网边界。

替代方案：直接把 `RA2MagnetPoint` 接到主玩法角色。该方案短期反馈明显，但会把绳索手感绑定到布娃娃骨骼刚体，容易影响非布娃娃攀爬、IK 和移动端稳定性；MVP 先不采用。

### 3. 张力方向基于逻辑绳路局部段

对于直连绳路，张力方向从被约束玩家指向另一名玩家。对于经过铆钉的绳路，张力方向从被约束玩家指向它在逻辑绳路上的相邻约束点；如果被约束玩家跨过多个转折点，仍使用路径排序决定最近的相邻点。

选择原因：这能保留铆钉转折的玩法意义，使玩家被拉向当前绳段，而不是被错误拉向另一名玩家或全局中心。

替代方案：始终使用两名玩家直连方向。该方案简单，但会让铆钉转折失效，视觉绳路和真实反馈不一致。

### 4. 3C 消费反馈时分状态处理

角色 3C 只在攀爬/悬挂等非布娃娃主控阶段应用绳索反馈。应用顺序建议为：输入目标/躯干目标计算后，胶囊体防穿模前，对目标速度和目标位移做张力方向投影限制；随后再由现有防穿模处理墙体接触。布娃娃摔落阶段优先不做硬约束，MVP 只显示张力提示；如后续需要布娃娃绳索，可作为单独变更设计。

选择原因：这能最大化复用现有 3C 管线，避免和 Ragdoll Animator 2 或物理混合产生不可控冲突。

替代方案：布娃娃阶段也立即启用物理 Joint。该方案反馈更真实，但容易引入骨骼炸裂、穿墙和平台差异，超出本次 MVP。

### 5. 先在现有绳索测试关卡验证

实现顺序优先扩展 `RivetRopeVisualLab.scene` 或等价现有绳索测试关卡，让测试关卡在保持当前绳索视觉的前提下加入两个可移动测试端点、张力反馈调试值和一个可观察的角色/代理体。只有当测试关卡中的限距、阻尼和回弹手感可接受后，才把消费入口接入主玩法 Binder。

选择原因：绳索反馈是手感功能，直接接入主玩法会混入联网、角色状态、输入和场景复杂度。先在测试关卡验证能更快调参，也符合“视觉保持现在”的范围控制。

替代方案：直接在主玩法角色上调试。该方案更接近最终体验，但定位问题成本高，且容易把反馈模型问题误判为主玩法 Binder 或网络同步问题。

### 6. 联网只同步玩法事实，不同步绳索反馈帧

本地和远端客户端基于相同的铆钉事件、玩家端点和配置分别计算绳索反馈。协议不新增高频 `rope.force` 帧消息；如需要调试，可在本地 UI 显示张力值和原因。

选择原因：绳索反馈是端点和绳路的派生结果，高频同步会增加带宽和插值复杂度，也无法保证移动端一致手感。

替代方案：主机每帧广播张力状态。该方案能让远端观测更一致，但会提高协议复杂度，并与现有 relay 只转发轻量玩法事件的边界冲突。

## Risks / Trade-offs

- 反馈过硬导致角色卡墙或抖动 → 限制单帧最大修正量，在 3C 防穿模前应用，并为阻尼、回弹、最大张力提供配置。
- 反馈过软导致玩家感受不到绳索 → 在测试场景中显示张力强度、超出长度和速度修正，并提供调参预设。
- 铆钉转折路径排序错误导致方向异常 → 复用现有逻辑绳路点序，并为直连、单铆钉、多铆钉路径增加 EditMode 测试。
- 本地和远端表现略有差异 → 保持规则输入轻量确定，远端以表现一致为目标，不把远端视觉误差反向写入规则。
- FImpossible 式反馈过早接入布娃娃导致不可控 → MVP 只抽象为 3C 的柔性牵引/阻尼策略，先在测试关卡验证，不直接启用 RA2 磁吸点控制主玩法角色。
- 修改测试关卡时误改视觉目标 → 保持现有视觉组件和材质配置，只增加反馈调试端点、代理体和调试 UI。
- 布娃娃阶段缺少真实绳索拉扯 → 本次明确作为非目标，避免影响 MVP 稳定性；后续可单独设计物理 Joint 接入。

## Migration Plan

1. 在 `rivet-rope-system` 中新增反馈数据结构、配置和纯函数计算器，默认参数保持现有绳索行为基本不变，只有超过绳长时输出激活反馈。
2. 扩展现有绳索测试关卡，保持当前视觉绳索外观，只增加反馈调试端点、代理角色/测试体和张力调试 UI。
3. 在测试关卡中先验证直连、单铆钉和多铆钉路径下的限距、阻尼、轻回弹和 FImpossible 式柔性牵引手感。
4. 在 `climb-character-3c` 中接入反馈消费入口，通过配置开关启用实际运动限制和反馈。
5. 测试关卡手感可接受后，再将主玩法 Binder 接上本地/远端端点速度和绳索反馈消费。
6. 若反馈导致主玩法不可控，可通过配置关闭 3C 消费，仅保留绳索系统计算和测试关卡调试作为回滚策略。

## Open Questions

- 端点速度优先由 3C 提供，还是由绳索系统根据连续帧端点差分缓存？MVP 倾向由 3C 提供，测试中可直接构造。
- 测试关卡中的代理体先用 Transform 运动还是 Rigidbody 运动？MVP 倾向先用可控 Transform/速度代理，避免一开始被物理步进噪声干扰。
- 布娃娃摔落时是否需要绳索硬拉？建议另开变更处理，以免和当前稳定攀爬反馈耦合。
