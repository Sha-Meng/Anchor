## ADDED Requirements

### Requirement: 系统验证测试场景职责

系统验证测试场景 MUST 作为独立的 Unity/Tuanjie 测试场景，用于验证关卡抓点系统、受力系统、调试 UI 和移动端震动反馈的集成行为。该场景 MUST NOT 作为正式角色 3C、正式关卡流程、铆钉绳索伤害或联网同步的交付边界。

#### Scenario: 独立验证场景

- **WHEN** 开发者打开系统验证测试场景
- **THEN** 场景 MUST 提供可运行的抓点查询与受力评估验证环境，并 MUST NOT 依赖完整角色 3C 才能执行基础验证

#### Scenario: 不改变正式玩法边界

- **WHEN** 测试场景输出稳定、脱力或坠落相关调试结果
- **THEN** 这些结果 MUST 来自受力系统或其测试适配器链路，并 MUST NOT 在测试场景中新增一套正式玩法判定规则

### Requirement: 仿 Gym 场景构成

系统验证测试场景 MUST 参考 Gym 的攀爬平面、摄像机观察方式和 `Anchor Point` 数据组件摆放方式，提供可用于验证的有效抓点、低稳定抓点和无抓点区域。场景 SHALL 为后续障碍点和假点样例保留扩展空间。

#### Scenario: 有效抓点样例

- **WHEN** 场景初始化完成
- **THEN** 场景 MUST 包含至少一组可被关卡抓点系统注册并查询到的有效抓点样例

#### Scenario: 低稳定和空区域样例

- **WHEN** 开发者在场景中移动测试输入点
- **THEN** 场景 MUST 能覆盖低稳定值抓点和无抓点空区域，以验证稳定值变化和空结果行为

#### Scenario: 参考 Gym 数据组件

- **WHEN** 场景中的抓点参与关卡查询
- **THEN** 抓点 MUST 使用与 Gym 兼容的抓点数据组件或等效数据源，避免测试场景使用与正式场景完全不同的数据格式

### Requirement: 临时输入点控制

系统验证测试场景 MUST 在角色系统未完成前提供临时输入点控制能力。PC 端 MUST 支持鼠标点击或拖动输入；移动端 MUST 支持触摸点击或拖动输入。输入控制 MUST 将屏幕坐标映射为测试攀爬平面上的世界坐标，并分别表达左手和右手输入点状态。

#### Scenario: PC 鼠标输入

- **WHEN** 开发者在 PC 端点击或拖动测试场景
- **THEN** 系统 MUST 更新对应测试手的输入点世界坐标，并将该坐标用于后续关卡查询和受力评估

#### Scenario: 移动端触摸输入

- **WHEN** 玩家在移动端触摸或拖动测试场景
- **THEN** 系统 MUST 将触摸位置映射为测试输入点世界坐标，并持续刷新对应手的关卡查询和受力评估

#### Scenario: 未命中攀爬平面

- **WHEN** 输入屏幕坐标无法映射到测试攀爬平面
- **THEN** 系统 MUST 将对应手标记为未接触或无效输入，并 MUST NOT 用旧坐标继续产生有效受力

### Requirement: 关卡定位实时验证

系统验证测试场景 MUST 实时调用关卡抓点查询能力，显示并使用输入点附近的抓点事实。查询结果 MUST 至少包含是否命中、抓点标识或调试名、距离、稳定值和空结果状态。

#### Scenario: 输入点命中有效抓点

- **WHEN** 测试输入点位于有效抓点查询范围内
- **THEN** 场景 MUST 显示命中的抓点信息、距离和稳定值，并将该抓点事实提供给受力输入适配器

#### Scenario: 输入点位于无抓点区域

- **WHEN** 测试输入点位于无抓点区域
- **THEN** 关卡查询结果 MUST 明确表示未命中或稳定值为 `0`，并 MUST 将空结果提供给受力输入适配器

#### Scenario: 拖动跨越抓点边缘

- **WHEN** 开发者将输入点从抓点中心拖动到抓点有效半径外
- **THEN** 场景显示的距离、稳定值和命中状态 MUST 随输入位置实时变化

### Requirement: 受力分析实时验证

系统验证测试场景 MUST 将左右手输入点、关卡查询结果、临时耐力或默认耐力状态组装为受力输入快照，并实时调用受力系统输出分析结果。输出 MUST 至少显示整体受力状态、左右手是否有效受力、主要失败原因和是否触发坠落。

#### Scenario: 双手有效抓点稳定

- **WHEN** 左右测试输入点均命中满足受力阈值的有效抓点，且临时耐力状态允许受力
- **THEN** 受力分析结果 MUST 输出稳定状态，并标记左右手均有效受力

#### Scenario: 单手无效

- **WHEN** 一只测试手命中有效抓点而另一只测试手位于无抓点区域或低于最小抓附阈值
- **THEN** 受力分析结果 MUST 输出对应的单手脱力或单手无效状态，并显示主要失败原因

#### Scenario: 双手无效

- **WHEN** 左右测试输入点均位于无抓点区域、稳定值不足或被标记为无效接触
- **THEN** 受力分析结果 MUST 输出双手脱力或坠落相关状态，并在满足坠落触发条件时显示坠落事实

#### Scenario: 状态缓冲验证

- **WHEN** 输入点短暂离开有效抓点后在受力系统配置的缓冲时间内恢复
- **THEN** 场景 MUST 显示状态切换过程，并 MUST 允许验证受力系统的缓冲规则是否生效

### Requirement: PC 调试 UI

系统验证测试场景 MUST 在 PC 端提供调试 UI，用于展示输入、关卡查询、受力分析和关键参数。UI MUST 支持每帧刷新核心字段，并 SHOULD 避免通过高频 Console 日志作为主要观察方式。

#### Scenario: 显示输入和关卡结果

- **WHEN** 开发者在 PC 端移动测试输入点
- **THEN** UI MUST 显示左右手输入点坐标、是否接触测试平面、命中抓点、距离和稳定值

#### Scenario: 显示受力结果

- **WHEN** 受力系统完成一帧评估
- **THEN** UI MUST 显示整体状态、左右手有效性、失败原因和最近一次状态切换

#### Scenario: 显示空结果

- **WHEN** 输入点没有命中任何有效抓点
- **THEN** UI MUST 明确显示空结果或稳定值 `0`，避免开发者误判为 UI 未刷新

### Requirement: 移动端震动反馈

系统验证测试场景 MUST 在移动端提供震动反馈适配。未触摸、未命中有效抓点、稳定值为 `0` 或对应手无效受力时 MUST 不震动或立即停止震动；命中有效抓点并可受力时，震动强度、脉冲频率或持续时间 MUST 随受力稳定程度增强。

#### Scenario: 无抓点不震动

- **WHEN** 移动端触摸点位于无抓点区域或关卡查询稳定值为 `0`
- **THEN** 系统 MUST 不触发震动，并 MUST 停止该触摸先前产生的持续或循环震动

#### Scenario: 稳定越高震动越强

- **WHEN** 移动端触摸点命中有效抓点，且受力系统判定对应手可受力
- **THEN** 系统 MUST 根据稳定值或受力稳定程度提高震动反馈强度、脉冲频率或持续时间

#### Scenario: 无效受力停止震动

- **WHEN** 触摸点虽然接近抓点但受力系统判定对应手无效受力
- **THEN** 系统 MUST 不输出表示稳定的震动反馈，并 MUST 显示或记录主要无效原因以便调试

#### Scenario: 触摸结束停止震动

- **WHEN** 移动端触摸结束、取消或离开测试区域
- **THEN** 系统 MUST 停止该触摸对应的震动反馈，避免震动残留

### Requirement: 稳定性和性能验收

系统验证测试场景 SHALL 支持高频点击和拖动输入下的稳定运行。运行时 MUST 避免空引用异常、明显卡顿、异常日志刷屏和震动状态卡死；查询和 UI 刷新 SHALL 复用现有关卡系统与受力系统的高频调用边界。

#### Scenario: 高频拖动输入

- **WHEN** 开发者或玩家持续快速拖动一个或两个测试输入点
- **THEN** 场景 MUST 持续刷新关卡查询、受力分析、PC UI 或移动端反馈，并 MUST NOT 出现空引用异常或持续错误日志

#### Scenario: 场景重复进入

- **WHEN** 开发者多次进入、退出或重新播放系统验证测试场景
- **THEN** 关卡查询、受力评估、UI 和震动反馈 MUST 正确初始化和清理，并 MUST NOT 保留上一轮测试的无效状态

#### Scenario: 反馈状态不卡死

- **WHEN** 输入从高稳定抓点快速移动到无抓点区域或触摸被取消
- **THEN** UI 和移动端震动 MUST 在后续刷新中反映无效状态，并 MUST NOT 继续显示或输出高稳定反馈
