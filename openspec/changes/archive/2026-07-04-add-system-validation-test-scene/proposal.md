## Why

关卡抓点系统和受力系统已经分别定义，但角色 3C 尚未完成，缺少一个能在真实场景中快速验证两者接口、数值和稳定性的基础测试环境。现在需要仿照 Gym 场景构造一个专用测试场景，用手指点击/拖动或鼠标输入直接代表左右手输入点，先验证关卡定位、抓点查询、受力分析和反馈映射是否正确。

## What Changes

- 新增一个仿 Gym 的系统验证测试场景，用于摆放可攀附点、低稳定点、空区域和后续可扩展的障碍/假点样例。
- 新增测试输入模式：在角色系统未完成前，PC 端使用鼠标点击/拖动或等效调试输入，移动端使用触摸点击/拖动，直接驱动左右手输入点坐标。
- 在测试场景中接入关卡抓点查询和受力评估链路，实时计算命中抓点、稳定值、左右手有效性、整体受力状态和失败原因。
- PC 端提供调试 UI，显示输入点坐标、最近抓点、稳定值、受力结果、状态变化和关键参数。
- 移动端接入震动反馈：未触碰或未命中抓点时不震动；命中有效抓点后，受力越稳定震动越强烈。
- 明确该场景用于功能正确性和稳定性验证，不作为最终角色 3C、完整关卡玩法或正式 UI 的交付。

## Capabilities

### New Capabilities

- `system-validation-test-scene`: 定义仿 Gym 的系统验证测试场景、临时输入、调试 UI、移动端震动反馈，以及关卡抓点系统和受力系统的集成验收要求。

### Modified Capabilities

- 无。

## Impact

- 预计影响 `Assets/DesignerSpace/` 或 `Assets/Anchor/` 下的测试场景、调试输入脚本、调试 UI、移动端震动反馈适配和系统集成脚本。
- 依赖现有或正在实现的 `level-anchor-system` 查询能力，以及 `force-system` 的受力输入快照、评估器和调试输出。
- 不引入新的第三方依赖；优先使用 Unity/Tuanjie 现有 UI、输入和移动设备震动能力。
- 验收重点是关卡定位、抓点稳定值、受力分析结果和反馈映射在高频输入下保持正确、可视、可调和稳定。

## Acceptance Criteria

- 在测试场景中点击/拖动输入点时，系统能实时显示输入点附近的抓点查询结果、稳定值和受力判定结果。
- 输入点位于无抓点区域时，关卡查询返回空或稳定值为 `0`，受力结果不会被误判为稳定，移动端不触发震动。
- 输入点越接近高稳定有效抓点，PC 调试 UI 显示的稳定值和受力稳定程度越高，移动端震动强度越强。
- 场景能覆盖稳定攀附、低稳定抓点、单手失效、双手失效、空区域和状态切换缓冲等基础路径。
- 高频拖动输入不应导致明显卡顿、异常日志刷屏、空引用异常或震动持续卡死。
