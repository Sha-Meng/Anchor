## Context

Anchor 当前已有 Gym 场景雏形，并已提出 `level-anchor-system` 与 `force-system` 两个基础能力：前者负责收集 `Anchor Point` 并提供抓点事实查询，后者消费输入快照并输出稳定、脱力、坠落等受力状态。角色 3C 尚未完成，导致两套系统缺少一个低成本、可重复、可视化的联调环境。

本设计新增一个仿 Gym 的系统验证测试场景。它用临时输入点代替角色左右手，让开发者在 PC 上通过鼠标/调试 UI、在移动端通过触摸拖动，直接验证关卡定位、抓点稳定值、受力评估和震动反馈的正确性与稳定性。

## Goals / Non-Goals

**Goals:**

- 提供一个独立测试场景，用于验证关卡抓点查询和受力分析链路。
- 在角色系统完成前，用点击/拖动输入点代表左右手世界坐标。
- 在 PC 端通过 UI 和可视化标记显示查询结果、受力结果和失败原因。
- 在移动端根据受力稳定程度输出震动反馈：无有效抓点不震动，越稳定震动越强。
- 支持高频拖动输入下的稳定性验证，避免空引用、卡顿、震动残留和日志刷屏。

**Non-Goals:**

- 不实现正式角色 3C、布娃娃、手部 IK、镜头和动画表现。
- 不替代 Gym 主场景，不要求制作完整关卡流程或最终美术。
- 不改变 `level-anchor-system` 和 `force-system` 的职责边界。
- 不实现铆钉库存、绳索保护、生命扣除或联网同步。
- 不承诺所有移动设备都支持连续可调震动；硬件能力不足时允许退化为脉冲频率或时长映射。

## Decisions

### Decision 1: 新建独立验证场景，参考 Gym 布局和数据组件

实现阶段应新建 `SystemValidationTest` 或同等命名的独立场景，复用 Gym 的攀爬平面、摄像机观察方式和 `Anchor Point` 数据组件摆放习惯。场景中应有少量高稳定抓点、低稳定抓点、空区域，并为后续障碍点、假点预留样例区域。

理由：直接改 Gym 会把验证脚本、临时 UI 和正式玩法场景混在一起，影响后续主场景迭代。独立场景能保持快速试验，同时通过复制 Gym 的结构验证真实接入方式。

备选方案：在 Gym 中加调试模式。该方案接入更真实，但会把未完成的角色系统、主场景对象和验证工具互相耦合，不适合当前阶段。

### Decision 2: 输入点通过屏幕射线映射到攀爬平面

测试输入控制器接收鼠标或触摸坐标后，从测试摄像机发射射线，与场景配置的攀爬平面、墙面 Collider 或显式测试 Plane 相交，得到左右手输入点世界坐标。PC 端可使用左键/右键、键盘切换或 UI 按钮指定左/右手；移动端默认使用屏幕左半区控制左手、右半区控制右手，多点触控时分别追踪对应触点。

若射线未命中测试平面，该手输入应标记为未接触，并向受力适配器提交无效接触事实。

理由：该方式贴近未来手部贴墙输入，但不依赖角色模型、IK 或物理组件。它也能同时支持鼠标和触摸。

备选方案：直接使用屏幕坐标或 UI 坐标评估抓点。该方案实现快，但无法验证关卡系统的世界坐标查询和 Gym 类场景布局。

### Decision 3: 使用测试适配器组装受力输入快照

验证场景应提供 `ValidationForceInputAdapter` 或同等组件，把左右手输入点、关卡抓点查询结果、临时耐力值和接触状态组装为受力系统输入快照。受力评估器仍只消费快照，不直接读取场景、输入设备或 UI。

理由：测试场景需要绕过未完成的角色 3C，但不能绕过受力系统本身。适配器能模拟未来角色系统的接入边界，也便于后续替换为真实角色输入。

备选方案：在测试脚本中手写稳定/脱力判定。该方案会产生第二套规则，无法验证 `force-system` 的真实行为。

### Decision 4: PC 调试 UI 显示一帧链路的关键事实

PC 调试 UI 至少显示左右手输入点、是否接触测试平面、最近抓点 ID/名称、距离、抓点稳定值、左右手有效受力、整体受力状态、失败原因、最近状态切换和关键参数。场景中还应使用 Gizmo、LineRenderer 或简单图标标记输入点、命中抓点和查询半径。

理由：该测试场景的主要价值是验证正确性。只看角色表现或日志不足以定位是输入映射、关卡查询还是受力规则出了问题。

备选方案：只输出 Console 日志。该方案容易刷屏，且在高频拖动时难以观察连续变化。

### Decision 5: 移动端震动反馈由稳定程度映射，且必须可停止

移动端反馈层读取受力结果和抓点稳定值，只有当输入点命中有效抓点且对应手可受力时才触发震动。震动强度应随稳定程度增加而增强；若平台 API 不支持幅度控制，可用脉冲频率、持续时间或间隔近似表达强弱。无触摸、未命中抓点、稳定值为 `0`、障碍/假点失效或手部无效受力时必须停止或不触发震动。

理由：用户明确希望“没有触碰到抓点不震动，受力越稳定震动越强烈”。将震动放在反馈适配层中，可以验证手感映射，同时不污染受力系统核心规则。

备选方案：直接按距离最近抓点震动。该方案能帮助摸索，但会在不可受力、低耐力或假点失效时给出误导反馈。

### Decision 6: 验证场景优先覆盖确定性路径，再扩展交互压力测试

首批样例应覆盖：双手高稳定抓点、单手空区域、双手空区域、低稳定值抓点、拖动跨越抓点边缘、状态缓冲恢复和持续失效触发坠落。障碍点与假点可在底层规格支持后加入同一场景的扩展区域。

理由：场景验证应先保证关卡定位和受力链路正确，再验证复杂内容类型。确定性路径也更容易转化为手工测试清单和后续自动化测试。

备选方案：一次性制作完整关卡。该方案展示效果更好，但会把关卡内容制作成本提前到系统验证阶段。

## Risks / Trade-offs

- [Risk] 输入射线与实际未来手部吸附逻辑不一致 → Mitigation：把测试输入明确标记为临时适配器，后续真实角色 3C 接入时只替换输入来源。
- [Risk] 移动设备震动 API 支持差异导致强度映射不一致 → Mitigation：封装 `IHapticFeedbackAdapter` 或同等接口，并提供幅度、脉冲频率、关闭三种策略。
- [Risk] 测试场景和 Gym 逐渐偏离 → Mitigation：只复制最小结构和数据组件，保留场景说明，并在任务中加入与 Gym 布局对照检查。
- [Risk] 调试 UI 每帧分配或日志刷屏影响稳定性判断 → Mitigation：UI 使用固定字段刷新，日志加节流，查询热路径避免临时分配。
- [Risk] 受力系统或关卡系统尚未完全实现导致场景无法一次接通 → Mitigation：按接口边界提供可替换的测试数据源或空实现，但任务验收必须最终接入真实能力。

## Migration Plan

1. 新建系统验证测试场景，复制或参考 Gym 的攀爬平面、摄像机和抓点摆放方式。
2. 添加测试输入控制器，将 PC 鼠标和移动端触摸映射为左右手世界坐标。
3. 接入关卡抓点查询，显示最近抓点、稳定值和空结果。
4. 接入受力输入适配器和受力评估器，显示整体状态、左右手有效性和失败原因。
5. 添加 PC 调试 UI、场景可视化标记和基础手工验收路径。
6. 添加移动端震动反馈适配，并验证无抓点不震动、稳定越高反馈越强。

Rollback 策略：若验证场景影响主工程运行，可保持变更完全隔离在测试场景和调试脚本中；必要时从 Build Settings 移除该场景，不影响 Gym 或正式玩法场景。

## Open Questions

- PC 端左右手输入的默认方案是左/右鼠标键，还是用 UI Toggle/快捷键切换当前手？
- 移动端震动需要区分左右手反馈吗？当前 MVP 可先用单一设备震动表达“当前触摸点稳定程度”。
- 障碍点和假点是否要在第一版测试场景中落地，还是等关卡系统支持点位类型后再扩展？
