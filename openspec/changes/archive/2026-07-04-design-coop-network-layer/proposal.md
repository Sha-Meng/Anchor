## Why

Anchor 计划先做双人合作联网攀岩手游的 MVP，需要用最小服务器能力验证“两台客户端能进入同一房间并看到彼此角色状态”。现阶段优先降低实现成本和运维复杂度，服务器只承担房间管理与消息转发，不做完整玩法模拟、扩容规划或复杂弱网恢复。

## What Changes

- 定义 MVP 网络层总体方案，包括轻量服务器、房间流程、客户端自权威状态同步和基础部署方式。
- 明确服务器作为 relay 转发端：不计算攀岩物理，不判定角色移动，不维护复杂会话模拟。
- 明确每个客户端计算并上报自己角色的权威状态，另一端只接收并平滑显示远端角色。
- 明确 MVP 玩家身份策略：服务器临时 `playerId`、客户端本地 `clientId`，不要求创建角色或设置昵称，不直接使用设备 ID。
- 明确统一协议信封和 JSON payload 配置扩展方式，具体攀岩同步内容后续随玩法自定义。
- 明确修改 `game.*` JSON payload 配置不需要重新部署服务器。
- 生成工程协议规范文档和项目 skill，指导后续协议维护、客户端配置和服务器部署。
- 明确当前房间列表、创建房间、加入房间、房主开始和双方进入游戏的 MVP 交互流程。
- 将工程内基础 demo 场景端到端流程作为本次任务验收标准，覆盖连接、建房、加入、开局、进入游戏和自定义协议收发。
- 规划腾讯云服务器 `43.156.16.10` 作为 MVP 联机测试入口，优先验证公网可达、房间创建和双人同步。
- 暂不处理服务器扩容、复杂弱网、断线重连、商业级安全和反作弊；这些能力仅在 MVP 证明玩法可行后再评估。

## Capabilities

### New Capabilities

- `coop-networking`: 双人合作攀岩 MVP 的轻量房间、消息转发、客户端自权威状态同步和腾讯云测试部署能力。

### Modified Capabilities

- 无。

## Impact

- 影响 Unity/Tuanjie 客户端的网络抽象、房间流程、本地角色状态上报和远端角色表现同步。
- 影响后端服务边界，包括房间管理、连接管理、消息广播和最小日志。
- 影响部署策略，包括腾讯云单机测试部署、端口规划和服务可达性验证。
- 可能引入轻量传输库、简单消息协议、后端运行时和启动脚本，但本变更阶段仍只产出设计与任务。
