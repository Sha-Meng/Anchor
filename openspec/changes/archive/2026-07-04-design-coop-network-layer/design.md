## Context

Anchor 当前目标是双人合作联网攀岩手游的 MVP。这个阶段最重要的不是搭建完整权威服务器，而是尽快验证核心体验：两名玩家能进入同一房间，各自操作自己的攀岩角色，并在对方屏幕上看到相对平滑、可理解的远端角色状态。

用户已有腾讯云服务器 `43.156.16.10`。当前连通性检查显示该机器公网可达，`80` 端口有 nginx 返回，`22` 和 `443` 端口 TCP 可连。MVP 可以先把它作为轻量 relay 服务器使用。

## Goals / Non-Goals

**Goals:**

- 用最小服务器能力支撑双人房间和消息转发。
- 每个客户端计算自己角色的状态，并把自己的角色状态作为该角色的权威状态广播给对方。
- 远端客户端只负责接收、插值和平滑显示另一个玩家的角色状态。
- 使用可扩展的统一协议信封先跑通 Editor/真机双端联机，具体玩法同步内容后续按玩法自定义。
- 支持房间列表、创建房间、加入房间、房主开始和两端进入游戏的完整 MVP 流程。
- 服务器部署尽量轻量，能启动、能收发、能维护当前房间列表、能看基础日志即可。
- 明确 MVP 玩家身份来源，不要求创建角色流程，也不直接依赖设备 ID。
- 前端需要能通过 JSON 协议定义配置玩法同步 payload，并生成或维护工程协议规范文档。

**Non-Goals:**

- 不做服务器端攀岩物理模拟。
- 不做服务器权威判定抓握、坠落、机关、胜负或反作弊。
- 不做服务器扩容、多实例调度、Redis、灰度发布或复杂运维。
- 不做完整弱网、断线重连、会话恢复或后台恢复。
- 不做正式账号体系、商业级安全和版本兼容策略。
- 不在设计阶段固定攀岩玩法的全部同步字段；协议只规定外层格式和扩展方式。

## Decisions

### 1. 网络拓扑：采用轻量客户端-服务器 relay

推荐架构：

- 客户端 A：计算玩家 A 的完整角色状态，发送给服务器。
- 客户端 B：计算玩家 B 的完整角色状态，发送给服务器。
- Relay 服务器：维护房间内的两个连接，把每个客户端的消息转发给同房间另一个客户端。

服务器需要知道：

- 房间 ID。
- 连接 ID。
- 服务器分配的 `playerId`。
- 客户端上报的可选 `clientId`。
- 当前房间列表。
- 房间状态：等待中、游戏中、已关闭。
- 房主连接。
- 房间内最多两个玩家。
- 每个玩家最近一次上报时间。
- 基础日志和错误原因。

服务器不需要知道：

- 角色攀岩物理细节。
- 抓握点是否合法。
- 体力、绳索、坠落和胜负规则。
- 关卡内部状态机。

选择原因：

- MVP 的目标是验证合作体验和网络同步手感，服务器纯转发实现最快。
- 双人房间规模很小，服务器不做规则计算也能满足早期可玩验证。
- 客户端自权威可以直接复用本地单人玩法逻辑，减少对玩法代码的侵入。

主要代价：

- 客户端可以作弊或上报不合理状态，但 MVP 阶段可以接受。
- 双方对协作机关、胜负和碰撞的理解可能不一致，需要设计上减少强一致依赖。
- 如果未来要上线或做竞技/排行榜，需要迁移到服务器校验关键规则。

### 2. 传输协议：MVP 优先 WebSocket

MVP 推荐先使用 WebSocket：

- 连接简单，Unity 客户端和轻量后端都容易实现。
- 可以同时承载房间消息、状态同步、心跳和调试日志。
- 走 `80` 或后续配置好的 `443`，更容易穿过普通网络环境。
- 调试成本低，可以直接查看 JSON 或 MessagePack 消息。
- 房间流程和游戏转发可以先共用同一条连接，减少客户端状态管理成本。

不建议 MVP 一开始使用 KCP/ENet：

- 需要额外验证 UDP 安全组、移动网络、库兼容和断线状态。
- 对纯转发状态同步来说，WebSocket 的性能足够支撑双人 MVP。
- 可靠 UDP 的收益主要体现在更低延迟和更强弱网控制，当前不是第一优先级。

建议消息编码：

- 第一版使用 JSON，便于调试。
- 如果帧率和包体成为问题，再切换 MessagePack 或二进制结构。

HTTPS/WSS 决策：

- MVP 内部测试阶段可以先不部署 HTTPS，使用 `ws://43.156.16.10:<port>` 或 nginx 反代的明文 WebSocket。
- 不部署 HTTPS 的主要影响是传输内容明文可见，不能承载账号密码、正式 token、支付或隐私数据。
- Unity/Tuanjie 原生客户端不受浏览器 HTTPS 页面禁止连接明文 WebSocket 的限制；如果未来做 WebGL 或网页大厅，就需要 WSS。
- 正式外测、上架、账号登录或开放公网长期使用前，应配置域名和 TLS，并改用 `wss://`。
- 仅使用 IP 地址做 HTTPS 不理想，证书通常需要绑定域名；当前阶段没有明显收益，可以暂不处理。

### 3. 玩家身份：服务器临时 playerId + 客户端本地 clientId

MVP 不需要创建角色流程，也不建议直接使用设备 ID。推荐使用两层身份：

- `playerId`：服务器在 WebSocket 连接建立后临时分配，例如 `p_8f3a2c`。它用于当前连接、房间房主、房间玩家列表和消息路由。
- `clientId`：客户端首次启动时生成 UUID 并保存在本地，例如 Unity `PlayerPrefs`。它只用于调试和后续可选的轻量重连，不作为安全身份。

昵称策略：

- MVP 不提供昵称设置交互。
- 房间界面可以直接显示 `playerId`、`host` / `guest`，或由客户端本地随机生成展示名。
- 展示名不进入核心协议，不作为房主、房间、同步或权限判断依据。

不推荐直接使用设备 ID：

- 移动平台获取设备 ID 可能涉及权限、隐私合规和平台差异。
- 设备 ID 不适合作为公开传输的玩家标识，也不适合作为安全凭证。
- MVP 不需要持久账号，使用设备 ID 会过早引入隐私和兼容成本。

是否需要创建角色流程：

- MVP 不需要。连接服务器后自动获得 `playerId`，创建/加入房间即可。
- 如果后续需要角色外观、存档或长期账号，再独立设计账号/角色系统，不和 relay 协议绑定。

身份流程：

1. 客户端启动时读取本地 `clientId`；如果不存在，则生成 UUID 并保存。
2. 客户端连接 relay 时发送可选 hello 信息，包含 `clientId` 和客户端版本。
3. 服务器返回 `system.welcome`，分配本次连接使用的 `playerId`。
4. 后续房间和游戏消息中的 `senderId` 以服务器记录的 `playerId` 为准，客户端传来的 `senderId` 仅用于调试，不作为可信来源。

### 4. 协议框架：统一信封，payload 可自定义

协议外层使用统一消息信封，服务器只强依赖房间控制字段和路由字段。玩法同步数据放在 `payload` 内，后续可按攀岩机制自由扩展。

建议 JSON 信封：

```json
{
  "type": "game.state",
  "requestId": "optional-client-request-id",
  "roomId": "AB12",
  "senderId": "player-1",
  "seq": 12,
  "sentAt": 123456.78,
  "schema": "player-state.v1",
  "payload": {}
}
```

字段约定：

- `type`：消息类型，使用命名空间，例如 `room.create`、`room.join`、`room.list`、`room.start`、`game.state`、`game.event`、`system.error`。
- `requestId`：可选，请求/响应配对用，客户端生成即可。
- `roomId`：房间相关消息必须携带；连接前或创建房间请求可以为空。
- `senderId`：服务器在连接或入房后分配，客户端发送时可携带，服务器转发时以服务器记录为准。
- `seq`：发送方递增序号，用于远端丢弃旧状态或做调试。
- `sentAt`：发送方时间戳，用于延迟估算和插值。
- `schema`：payload 的玩法数据版本，例如 `player-state.v1`、`climb-event.v1`。
- `payload`：玩法自定义内容；服务器不理解也应能转发。

扩展规则：

- 新增玩法字段优先加到 `payload`，不要修改信封字段语义。
- 新玩法消息优先增加新的 `schema` 或新的 `type`，旧客户端无法识别时可以忽略。
- payload 字段默认向后兼容：新增字段应为可选，删除或改名需要升级 `schema`。
- 服务器只校验消息大小、JSON 格式、房间归属、房间状态和允许转发的 `type`，不校验攀岩玩法规则。
- 服务端需要保留未知 `payload` 原样转发，避免后续每加一个玩法字段都要改服务器。

减少后续服务器变更的边界：

- 服务器固定实现 `system.*` 和 `room.*` 控制消息。
- 服务器对 `game.*` 消息只做房间归属、房间状态、大小限制和频率限制，然后原样转发。
- 新增攀岩同步字段时，只改客户端协议定义和 payload 处理，不改服务器。
- 新增 `game.state`、`game.event` 的 schema 版本时，不需要改服务器。
- 只有新增房间规则、匹配、账号、重连或服务端校验时，才需要改服务器。

前端协议配置方式：

- 在 Unity/Tuanjie 客户端保留一个可编辑 JSON 协议定义配置，例如 `coop-network-protocol.config.json`。
- 协议定义记录消息类型、schema 名称、payload 字段说明、发送频率、是否需要转发、调试显示名称。
- 运行时代码使用统一 `NetworkMessage` 信封发送，payload 可由具体玩法模块序列化为 JSON 对象。
- 工程协议规范文档以 `docs/coop-network-protocol.md` 为事实来源；实现时可从同一份字段定义维护 C# 常量或 Inspector 配置，避免散落硬编码字符串。
- 修改 JSON 配置中的 `game.*` payload 字段、schema 示例、发送频率或调试显示名，不需要重新部署服务器；客户端重新加载配置或重新出包即可。
- 只有修改 `system.*` / `room.*` 控制流程、服务器校验规则、房间状态机或部署配置时，才需要修改并重新部署服务器。

示例：请求当前房间列表

```json
{
  "type": "room.list",
  "requestId": "req-001",
  "payload": {}
}
```

示例：服务器返回当前房间列表

```json
{
  "type": "room.list.result",
  "requestId": "req-001",
  "payload": {
    "rooms": [
      {
        "roomId": "AB12",
        "hostId": "player-1",
        "playerCount": 1,
        "maxPlayers": 2,
        "state": "lobby"
      }
    ]
  }
}
```

示例：玩法自定义状态

```json
{
  "type": "game.state",
  "roomId": "AB12",
  "seq": 34,
  "schema": "player-state.v1",
  "payload": {
    "position": { "x": 0.1, "y": 2.3, "z": 0.0 },
    "rotationY": 90,
    "custom": {
      "leftHandGripId": "hold-03",
      "rightHandGripId": null,
      "stamina": 0.8
    }
  }
}
```

这里的 `custom` 只是示例，不是固定要求。后续玩法可以换成更适合攀岩系统的数据结构。

### 5. 权威边界：每个客户端权威自己的角色

MVP 权威规则：

- 本机玩家的输入、抓握、释放、移动、动画参数和局部玩法状态由本机计算。
- 本机周期性上报自己的角色状态快照。
- 服务器不修改快照，只补充服务器接收时间或连接元信息后转发。
- 远端客户端接收另一个玩家的快照，并用插值或平滑移动表现。

MVP 可先同步的示例内容：

- playerId。
- sequence。
- clientTime。
- position。
- rotation。
- velocity 或 movementState。
- isGrabbing。
- grabPointId 或 grabPointPosition。
- animationState。
- simpleActionEvent，例如 wave、fall、finish、checkpoint。

这些字段不是协议硬约束，只是第一版调试时可用的 payload 示例。真正的同步内容应随攀岩玩法实现调整。

不建议同步内容：

- 每一帧完整骨骼。
- 大量物理接触点。
- 复杂 ragdoll 状态。
- 只有本地 UI 使用的临时数据。

### 6. 同步频率：低成本状态流

建议默认值：

- 本地角色状态上报：10-15Hz。
- 远端插值缓冲：100-200ms。
- 心跳或 ping：每 2-5 秒一次。
- 房间消息和一次性动作事件：发生时发送。

判断标准：

- 如果远端角色看起来明显卡顿，再提高到 20Hz。
- 如果移动端发热、流量或服务日志压力明显，再降频并增强插值。
- 如果攀岩动作需要瞬时反馈，把抓握、释放、坠落这类事件单独作为可靠事件发送，不只依赖下一帧快照。

### 7. 房间流程：列表、创建、加入、房主开始

服务器维护内存态当前房间列表即可。MVP 不需要数据库，服务重启后房间丢失可以接受。

房间状态：

- `lobby`：房间已创建，玩家在简单房间界面等待。
- `starting`：房主点击开始后，服务器已广播开始消息，客户端准备进入游戏。
- `inGame`：双方已进入游戏同步阶段。
- `closed`：房间关闭，不再出现在可加入列表。

房间数据：

- `roomId`：短房间码或唯一 ID。
- `hostId`：创建房间的玩家。
- `players`：当前连接玩家列表，最多两人。
- `state`：房间状态。
- `createdAt`：创建时间。
- `lastActiveAt`：最近活动时间。

MVP 流程：

1. 客户端连接 relay，服务器返回 `system.welcome`，包含 `playerId`。
2. 客户端发送 `room.list`，服务器返回当前可加入房间列表。
3. 房主点击创建房间，客户端发送 `room.create`。
4. 服务器创建 `lobby` 房间，将创建者设为 `hostId`，返回 `room.created`，客户端进入简单房间界面。
5. 第二名玩家点击房间列表或输入房间码，发送 `room.join`。
6. 服务器校验房间存在、未满员且仍为 `lobby`，成功后向双方广播 `room.updated`。
7. 房间满两人后，房主按钮可用；房主点击开始，发送 `room.start`。
8. 服务器校验发送者是房主、房间满两人且状态为 `lobby`，然后广播 `room.starting`。
9. `room.starting` 建议携带 `startAt` 或倒计时秒数，例如当前服务器时间后 2 秒，两个客户端收到后同时切换到游戏场景。
10. 客户端进入游戏场景后发送 `room.enteredGame`，服务器收到双方确认后将房间状态设为 `inGame`，并广播 `room.inGame`。
11. 游戏中双方发送 `game.state` 和 `game.event`，服务器按房间转发。
12. 任一玩家离开或断开，服务器广播 `room.peerLeft`，MVP 可以直接返回房间界面或结束游戏。

核心房间消息：

- `system.welcome`：服务器分配连接身份。
- `room.list` / `room.list.result`：拉取当前可加入房间列表。
- `room.list.updated`：房间创建、加入、离开、开始或进入游戏时，服务器向在线客户端广播当前房间列表。
- `room.create` / `room.created`：创建房间并进入房间界面。
- `room.join` / `room.joined`：加入房间并进入房间界面。
- `room.updated`：广播房间玩家数量、房主和状态。
- `room.start` / `room.starting`：房主开始，双方准备进入游戏。
- `room.enteredGame` / `room.inGame`：客户端确认进入游戏，服务器确认同步阶段开始。
- `room.leave` / `room.peerLeft`：离开房间或队友离开。
- `system.error`：房间不存在、房间已满、非房主开始、状态不允许等错误。

“同时进入游戏”的处理：

- MVP 不需要严格帧同步，只需要体验上同时切场景。
- `room.starting` 中携带 `startAt` 或 `countdownMs`，客户端收到后显示倒计时并切换场景。
- 如果一个客户端加载较慢，进入后仍按最新收到的远端状态显示；MVP 不做复杂加载等待。

暂不做：

- 自动匹配。
- 好友邀请链路。
- 观战。
- 中途换人。
- 断线恢复同一会话。

### 8. Demo 场景验收标准

本次任务的工程验收标准是提供基础网络 demo 场景流程，完整验证登录式连接、建房、加入房间、房主开局、进入游戏和基础自定义协议交互。

demo 场景建议：

- `NetworkDemoEntry`：网络 demo 入口，负责连接服务器、刷新房间列表、创建房间、加入房间。
- `NetworkDemoRoom`：简单房间界面，显示房间码、玩家数量、房主状态和开始按钮。
- `NetworkDemoGame`：demo 游戏场景，显示本地玩家和远端玩家，用 JSON 配置定义的 `game.state` 和 `game.event` 验证自定义协议收发。

验收流程：

1. 启动两个客户端实例，进入 `NetworkDemoEntry`。
2. 两个客户端连接 relay，并收到 `system.welcome` 和不同的 `playerId`。
3. 客户端 A 创建房间并进入 `NetworkDemoRoom`。
4. 客户端 B 通过房间列表或房间码加入同一房间。
5. 房间未满员时开始按钮不可用；满员后只有房主可以点击开始。
6. 房主点击开始后，双方收到 `room.starting` 并进入 `NetworkDemoGame`。
7. 双方发送 `room.enteredGame`，收到 `room.inGame` 后开始同步。
8. demo 场景通过 JSON 配置定义的 `game.state` payload 转发基础状态，并驱动远端表现。
9. demo 场景触发一个 JSON 配置定义的自定义 `game.event`，另一端能在 UI 或日志中看到。

验收通过条件：

- 不需要昵称设置或角色创建即可完成流程。
- 修改 `game.*` JSON 配置中的 payload 示例或发送频率后，不需要重新部署服务器。
- 服务器日志能看到连接、建房、加入、开始、进入游戏和消息转发记录。

### 9. 腾讯云部署：单进程 relay 即可

建议部署形态：

- 一个轻量后端进程，监听 WebSocket 端口。
- nginx 可继续监听 `80`，后续可反向代理到 WebSocket 服务。
- 如果 HTTPS 需要启用，需要单独配置证书；当前 `https://43.156.16.10` 的 TLS 握手失败，不能视为可用 HTTPS。

最小端口建议：

- `22/tcp`：SSH 管理，当前可连。
- `80/tcp`：HTTP/nginx，当前返回 `200 OK`。
- `443/tcp`：当前 TCP 可连但 HTTPS 握手失败，需要配置证书或确认服务类型。
- MVP WebSocket 服务端口：可以先用 `8080/tcp` 或由 nginx 反代到 `80` 路径。

推荐部署步骤：

1. SSH 登录 `43.156.16.10`，确认系统、CPU 架构、可用内存、已安装运行时和 nginx 配置目录。
2. 在服务器创建 relay 服务目录，例如 `/opt/anchor-relay`。
3. 上传或拉取 relay 服务代码，安装运行时依赖。
4. 本机先启动服务监听 `127.0.0.1:8080` 或 `0.0.0.0:8080`。
5. 腾讯云安全组放行测试端口；如果使用 nginx 反代，则公网只需放行 `80`。
6. 配置 nginx WebSocket 反代，例如 `/ws` 转发到 `http://127.0.0.1:8080`。
7. 从开发机执行 WebSocket 连接测试，确认能收到 `system.welcome`。
8. 使用 `systemd` 或简单进程管理脚本保持服务运行。
9. 查看日志，确认创建房间、加入房间、房主开始和转发消息均有记录。

部署指引原则：

- relay 服务应从环境变量或配置文件读取端口、日志等级、房间上限、单消息大小上限和允许转发的 `game.*` 前缀。
- nginx 只负责反代 WebSocket，不理解协议内容。
- systemd 只负责进程保活，不承担业务逻辑。
- 玩法协议变更应优先发生在客户端 `payload` 和协议规范文档中，不要求重启或修改服务器。
- 服务器需要修改的情况应限制在房间流程、连接管理、安全限制和部署配置变化。

nginx 反代后，客户端地址可以是：

```text
ws://43.156.16.10/ws
```

如果暂不配置 nginx，也可以直接使用：

```text
ws://43.156.16.10:8080
```

前提是腾讯云安全组和系统防火墙都放行该端口。

最小日志：

- 服务启动时间。
- 连接建立和断开。
- 创建房间和加入房间。
- 房间内转发消息数量。
- 非法房间码、房间满员、消息格式错误。

### 10. 可行性结论

“服务器纯转发 + 每个客户端权威自己角色”的思路对 MVP 可行，前提是玩法设计接受以下约束：

- 第一版不要设计强一致的物理对抗或复杂双人机关。
- 胜负、检查点、救援等结果可以先由触发者客户端上报，另一端接受显示。
- 远端玩家主要是“同步表现对象”，不要让远端角色直接驱动本地关键物理判定。
- 后续如果发现合作玩法必须强一致，再把少量关键事件迁移到服务器校验，而不是一开始就做完整服务器模拟。

## Risks / Trade-offs

- [Risk] 客户端自权威可能出现作弊或不合理状态 -> Mitigation: MVP 只做内部测试，后续上线前再补关键规则校验。
- [Risk] 双方对胜负或机关结果可能短暂不一致 -> Mitigation: MVP 先减少强一致机关，关键事件由触发方广播，另一端接受。
- [Risk] WebSocket 延迟高于 UDP -> Mitigation: 双人 MVP 先以可实现为主，远端角色使用插值平滑；必要时再升级传输。
- [Risk] 服务器纯转发无法处理断线恢复 -> Mitigation: MVP 断线直接提示队友离开或房间结束。
- [Risk] HTTPS 当前不可用 -> Mitigation: MVP 可先用 HTTP/WebSocket 验证，正式测试前再配置域名和 TLS。

## Migration Plan

1. 确认腾讯云服务器登录方式、系统环境和可用端口。
2. 定义统一消息信封和房间协议，确保玩法 payload 后续可自定义扩展。
3. 在服务器上部署最小 WebSocket relay，支持房间列表、创建房间、加入房间、房主开始、广播消息和基础日志。
4. 在 Unity/Tuanjie 中实现最小客户端连接器，支持连接、房间列表、创建房间、加入房间、房主开始和断开。
5. 在测试场景中放置两个玩家表现对象：本地对象由输入驱动，远端对象由网络快照驱动。
6. 两台设备或两个 Editor 实例连接同一房间，验证房主开始后双方能进入游戏并互相显示状态。
7. 根据手感调整同步频率、插值缓冲和需要单独发送的动作事件。
